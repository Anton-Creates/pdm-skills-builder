# Спецификация Финансового маркетплейса

Создать детальные требования к финансовому маркетплейсу или агрегатору (аналоги Банки.ру, Сравни.ру, Финуслуги). Скилл помогает продакту спроектировать платформу, которая агрегирует предложения десятков банков, страховщиков и брокеров, обеспечивает бесшовную отправку заявок и монетизируется без ущерба для доверия пользователей.

## Процесс

1. **Определи тип агрегируемых продуктов.** (Вклады, потребительские кредиты, ипотека, автострахование ОСАГО/КАСКО, микрозаймы).
2. **Спроектируй модель интеграции с партнерами (API банков).** Как маркетплейс получает актуальные ставки и передает заполненные анкеты? (Передача лидов - Lead Gen, прямая интеграция API/СМЭВ, или перенаправление пользователя на сайт банка).
3. **Опиши баланс монетизации и доверия.** Как устроена модель монетизации (оплата за клик CPC, оплата за одобренную заявку CPL, или комиссия за выданный продукт CPS)? Как ранжируются предложения — честно по выгоде для клиента или с приоритетом спонсируемых партнеров?
4. **Спроектируй воронку и анкету пользователя.** Как минимизировать количество заполняемых полей (например, интеграция с Госуслугами/ЕСИА для автозаполнения анкеты).
5. **Опиши комплаенс и юридическую архитектуру.** Согласия на БКИ, требования 152-ФЗ, требования Банка России к операторам финансовых платформ.
6. **Сохрани вывод** в текущей рабочей директории как `finmarket-spec-[название-продукта].md`.

## Формат вывода

```
## Спецификация финансового маркетплейса: [Название]

### 1. Продуктовая стратегия и Модель агрегации
- **Категория финансовых продуктов:** [например, агрегатор вкладов с онлайн-открытием]
- **Схема интеграции с банками:**
  - *White Label / Прямая интеграция:* оформление происходит внутри маркетплейса, банк получает готовую сделку.
  - *Lead Generation:* маркетплейс собирает контакты и передает банку в CRM.
  - *Referral:* маркетплейс просто перенаправляет на сайт банка по реферальной ссылке.

### 2. Монетизация и Ранжирование предложений
- **Модель монетизации:** [CPL (оплата за заполненную анкету), CPS (процент от выданного кредита/открытого вклада), или рекламная модель]
- **Алгоритм ранжирования (Fair Play):**
  - Как формируется выдача по умолчанию (по лучшей ставке, по вероятности одобрения для данного клиента, или с учетом партнерской комиссии).
  - Как помечаются рекламные/спонсируемые предложения (маркировка рекламы).

### 3. Воронка пользователя и Автозаполнение (ЕСИА / Госуслуги)
- **Флоу сравнения:** фильтры по параметрам (сумма, срок, требования к залогу) -> сравнение карточек продуктов.
- **Бесшовная анкета:** интеграция с Госуслугами (ЕСИА) для мгновенного подтверждения личности и автозаполнения паспортных данных, адреса регистрации и уровня дохода (через ПФР).
- **Мульти-заявка:** отправка одной анкеты сразу в 5-10 банков-партнеров. Как агрегируются и отображаются ответы (одобрено/отклонено/изменены условия).

### 4. Доверие пользователей и Безопасность (Trust & Safety)
- **Сверка данных:** защита от подмены условий (как гарантируем, что ставка на маркетплейсе совпадает со ставкой в договоре банка).
- **Безопасность транзакций:** если маркетплейс принимает деньги (например, для открытия вклада на Финуслугах через СБП) — интеграция с РФИ (Регистратором финансовых транзакций).

### 5. Метрики успеха маркетплейса
- **Match Rate:** % пользователей, которые нашли хотя бы одно подходящее предложение и отправили заявку.
- **Approval Rate:** % одобрений банками-партнерами по заявкам, отправленным с маркетплейса (качество лида для банка).
- **Take Rate / Revenue per Lead:** средний доход платформы с одной отправленной/одобренной заявки.
- **Drop-off на анкете:** на каком поле анкеты пользователи чаще всего уходят.

### 6. Юридические требования и Регулирование (ЦБ РФ)
- **Лицензирование:** требования ЦБ РФ к реестру операторов финансовых платформ (закон № 211-ФЗ).
- **Согласие на БКИ:** обязательный шаг подписания согласия через СМС/УКЭП для проверки кредитной истории в нескольких бюро.
```

## Правила

- Оспаривай спецификации, где пользователь должен заполнять 30 полей вручную. Финансовый маркетплейс обязан использовать интеграцию с Госуслугами (ЕСИА/СМЭВ) для автозаполнения.
- Монетизация не должна убивать доверие. Алгоритм ранжирования должен быть прозрачным для пользователя. Если спонсорские предложения выводятся выше, они должны быть явно промаркированы.
- Учитывай риск изменения условий банком. Маркетплейс должен регулярно обновлять данные о ставках (через API банков или парсинг), чтобы избежать обвинений в недостоверной рекламе.
- Пиши на русском языке.

## Метрики (Fintech / Lending)

### Outcome metric
**risk-adjusted profit, approved good customers, portfolio margin.** Главный результат и ценность.

### Input metrics
**KYC pass rate, approval rate by risk bucket, time-to-decision, utilization.** Управляемые рычаги outcome.

### Guardrails
**NPL 30/60/90, fraud loss, complaints, regulatory breaches, manual review overload.** Что нельзя ухудшить.

### Diagnostic metrics
**vintage analysis, PD/LGD, funnel by segment, false positives/negatives, channel quality.** Где искать причину.

### Instrumentation
**application_id, risk bucket, decision, KYC steps, repayment/vintage data, fraud flags.** Какие данные нужны.

### Decision rules
- Ship / Iterate / Kill

### Универсальное правило метрики
Если вы предлагаете метрику, ответьте на 5 вопросов:
1. **Кто владеет этой метрикой?**
2. **Как часто её смотрим?**
3. **Какие события её считают?**
4. **Какой порог решения?**
5. **Как её можно испортить или накрутить?**
