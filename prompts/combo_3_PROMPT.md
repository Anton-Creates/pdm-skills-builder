# СКИЛЛ: /prd

# Генератор PRD (Product Requirements Document)

Создать лаконичный, actionable спецификационный документ (PRD) объемом до 2 страниц. Шаблон заставляет продакта принять жесткие решения по целям, скоупу и метрикам, а также поддерживает специальный AI/ML-режим для фич на базе искусственного интеллекта.

## Процесс

1. **Разбери концепт.** Кто целевая аудитория? Какую проблему решаем?
2. **Определи характер фичи.** Если фича использует ИИ/ML (например, автогенерация смет, скоринг, ИИ-ассистент) — включи ML-раздел.
3. **Опиши ML требования (для AI-режима):**
   - *Метрики модели vs. Метрики бизнеса:* укажи целевые Precision/Recall/F1-score модели и свяжи их с бизнес-метриками (например, одобряемость, конверсия).
   - *Обучающая выборка:* источники данных для обучения и разметки.
   - *Флоу отката (Fallback Logic):* что происходит в интерфейсе, когда модель ошибается, галлюцинирует, возвращает низкий уровень уверенности (confidence score) или падает по таймауту (например, показ статической заглушки, ручной выбор, переключение на оператора).
4. **Сформулируй MVP скоуп и границы (What is Out of Scope).**
5. **Сохрани вывод** в текущей рабочей директории как `prd-[название-фичи].md`.

## Формат вывода

```
## Product Requirements Document (PRD): [Название фичи]

### 1. Проблема и Целевая аудитория
- **В чем проблема:** [описание боли пользователя с примером]
- **Кто страдает:** [конкретный сегмент пользователей, а не абстрактное «все»]
- **Как решается сейчас:** альтернативные обходные пути пользователей.

### 2. Описание решения и Сценарии (Scope)
- **Что мы строим:** [суть решения, ключевая ценность]
- **Что мы НЕ строим (Out of Scope):** [критические границы скоупа, отсекающие раздувание требований]
- **Пользовательские сценарии:** последовательный пошаговый флоу работы фичи.

### 3. Требования к AI / ML (ML Requirements)
*(Заполняется только для ИИ-фич)*
- **Задача модели:** [например, классификация сметных позиций]
- **Метрики качества модели:** [минимальный порог точности Precision / Recall / F1]
- **Сценарий отката (Fallback Logic):**
  - Если уверенность модели (confidence) < 80%: [например, не применяем автозаполнение, а просим пользователя заполнить вручную с подсветкой подсказки].
  - При таймауте ответа API модели (> 500мс): [переключаемся на синхронный бэкап-алгоритм поиска по регулярным выражениям].

### 4. Метрики успеха и KPI
- **Первичная бизнес-метрика:** [один главный показатель с целевой цифрой, например: рост конверсии в сделку на +3%]
- **Защитная метрика (Guardrail):** [метрика, которая не должна ухудшиться, например: время загрузки страницы не более 1.5 сек].

### 5. Риски и Открытые вопросы
- **Продуктовые риски:** [риск, что пользователи не поймут фичу; план митигации]
- **Технические риски:** [зависимости от внешних API, производительность бэкенда].
```

## Правила

- Раздел «Что мы НЕ строим» обязателен. PRD без жестких границ скоупа приводит к бесконечной разработке и срыву сроков.
- Для ИИ-фич логика Fallback обязательна. Нельзя запускать нейросети на проде без сценария поведения системы при ошибке модели.
- Метрики успеха должны содержать измеримые цели и цифры, а не абстрактные лозунги.
- Пиши на русском языке.

# СКИЛЛ: /user-stories

# Генератор пользовательских историй

Превратить описание фичи в полный набор user story, готовых к спринт-планированию. Каждая история независимо поставляема и имеет тестируемые критерии приёмки.

## Процесс

1. Прочитай и проанализируй описание фичи.
2. Определи эпик и разбей его на независимо поставляемые user story.
3. Для каждой истории напиши критерии приёмки в формате Given/When/Then.
4. Определи граничные случаи и негативные сценарии для каждой истории.
5. Опиши зависимости между историями и внешними системами.
6. Явно определи, что выходит за скоуп.
7. Сохрани результат в текущей рабочей директории как `stories-[название-фичи].md`.

## Формат вывода

### Описание эпика
- Название эпика
- Цель (одно предложение — что этот эпик даёт пользователю?)
- Контекст (почему сейчас, что инициировало эту работу)

### Пользовательские истории

Для каждой истории:

#### История [номер]: [короткое название]

**История:** Как [конкретный тип пользователя], я хочу [конкретное действие], чтобы [измеримый или наблюдаемый результат].

**Приоритет:** Must-have / Should-have / Nice-to-have

**Критерии приёмки:**
- Given [предусловие], When [действие], Then [ожидаемый результат]
- Given [предусловие], When [действие], Then [ожидаемый результат]

**Граничные случаи:**
- Что происходит при [нестандартном условии]?
- Что происходит при [ошибке]?
- Что происходит при [граничном значении]?

**Примечания:** Любые подсказки по реализации или контекст, который нужен команде.

### Зависимости
- Между историями (какие истории должны быть выполнены перед другими)
- Внешние зависимости (API, сервисы, команды, согласования)
- Технические предпосылки

### Вне скоупа
- Фичи или поведения, явно исключённые из эпика
- Смежная работа, которая может возникнуть в обсуждениях, но относится к другому эпику

## Правила

- Каждая история должна быть независимо поставляема. Если история не может быть поставлена без другой — либо объедини их, либо явно укажи зависимость.
- Критерии приёмки должны быть тестируемыми — QA-инженер должен проверить каждый без лишних вопросов.
- Всегда включай негативные сценарии: что происходит при ошибке, таймауте, невалидном вводе, отсутствии прав.
- Пользовательские истории пишутся с точки зрения пользователя, а не системы. «Система валидирует ввод» — не user story.
- Будь конкретен в типе пользователя. «Как пользователь» — слишком расплывчато. «Как администратор workspace с доступом к биллингу» — конкретно.
- Польза в «чтобы» должна быть реальной. «Чтобы использовать функцию» — замкнутый круг. «Чтобы добавлять новых участников без создания тикета в поддержку» — реальная польза.
- Если описание фичи слишком расплывчато — перечисли уточняющие вопросы перед написанием историй.
- Добавляй оценку размера истории (S/M/L) только если контекста достаточно для оценки.
- Пиши на русском языке.

# СКИЛЛ: /roadmap

# Генератор роадмапа

Превратить хаотичный список фич, приоритетов или целей в чёткий продуктовый роадмап. Не диаграмма Ганта — стратегический инструмент коммуникации, показывающий что делаем, в каком порядке и почему.

## Процесс

1. **Разбери входные данные.** Разбери список фич, инициатив или целей. Принимай inline-списки, вставленные бэклоги или пути к файлам.
2. **Определи стратегические темы.** Сгруппируй связанные элементы в 3-5 тем (например, «Онбординг», «Монетизация», «Стабильность платформы»). Каждый элемент должен принадлежать теме.
3. **Распредели по временным горизонтам.** Помести каждый элемент в Now (этот квартал), Next (следующий квартал) или Later (будущее / нужна валидация). Основывайся на зависимостях, усилиях и стратегическом приоритете — не только на срочности.
4. **Составь карту зависимостей.** Определи элементы, блокирующие или открывающие другие. Отметь элементы критического пути.
5. **Оцени усилия.** T-shirt sizing: S (< 2 недель), M (2-6 недель), L (6-12 недель), XL (> 12 недель).
6. **Напиши стратегическое обоснование.** Одно предложение на каждый временной горизонт, объясняющее почему такая последовательность имеет смысл.
7. **Сохрани вывод** в текущей рабочей директории как `roadmap-[контекст].md`.

## Формат вывода

### Роадмап: [Продукт/Команда/Контекст]
**Последнее обновление:** [дата]
**Горизонт планирования:** [например, Q2-Q4 2026]
**Допущение по ёмкости:** [если известно, например, 1 команда из 5 инженеров]

---

### Стратегические темы

| Тема | Описание | Ключевая метрика |
|------|----------|-----------------|
| [тема] | [одна строка] | [что сдвинется если тема успешна] |

---

### Now (Этот квартал)
**Обоснование:** [Почему эти элементы первые — одно предложение.]

| Приоритет | Элемент | Тема | Размер | Зависимости | Статус |
|-----------|---------|------|--------|-------------|--------|
| 1 | [элемент] | [тема] | S/M/L/XL | [блокер или «Нет»] | Не начато / В работе |

### Next (Следующий квартал)
**Обоснование:** [Почему эти элементы вторые — одно предложение.]

| Приоритет | Элемент | Тема | Размер | Зависимости | Уверенность |
|-----------|---------|------|--------|-------------|-------------|
| 1 | [элемент] | [тема] | S/M/L/XL | [что должно выйти раньше] | Высокая/Средняя/Низкая |

### Later (Будущее / Нужна валидация)
**Обоснование:** [Почему эти элементы отложены — одно предложение.]

| Элемент | Тема | Размер | Почему позже |
|---------|------|--------|-------------|
| [элемент] | [тема] | S/M/L/XL | [конкретная причина: нужны данные, заблокировано, низкий приоритет и т.д.] |

---

### Карта зависимостей
- [Элемент A] → блокирует [Элемент B] (должен выйти до начала B)
- [Элемент C] → открывает [Элемент D, Элемент E]

### Ключевые компромиссы
- **[Компромисс 1]:** [Что выбрал и от чего отказался. Явно.]
- **[Компромисс 2]:** [Что выбрал и от чего отказался.]

### Что НЕ входит в роадмап
- [Элемент намеренно исключён] — Причина: [почему]

## Правила

- Now/Next/Later — формат по умолчанию. Не используй даты или спринты если пользователь явно не просит.
- Каждый элемент должен принадлежать теме. Если элемент не вписывается ни в одну тему — поставь под сомнение, стоит ли ему быть в роадмапе.
- «Later» — не парковка. У каждого элемента в Later должна быть конкретная причина отсрочки — не просто «низкий приоритет».
- Зависимости должны быть явными. Если Элемент B не может начаться до выхода Элемента A — скажи об этом.
- Раздел «Что НЕ входит в роадмап» обязателен. Роадмап, который ни от чего не отказывается — не роадмап.
- Оценки размеров должны быть последовательными. Если два элемента оба «M» — они должны быть примерно одинакового усилия.
- Не перегружай Now. Если в Now более 5-6 элементов — оспорь это: это не план, это список желаний.
- Если входной список не имеет стратегического контекста — спроси какие 1-2 главные цели команды, прежде чем выстраивать последовательность. Порядок без стратегии — просто список.
- Пиши на русском языке.

# СКИЛЛ: /prioritize

# Приоритизация бэклога (Prioritize)

Превратить хаотичный бэклог в ранжированный, защищаемый список приоритетов. Скилл поддерживает классические фреймворки (RICE, ICE, MoSCoW), а также WSJF (для Agile/SAFe) и банковский комплаенс-режим (Regulatory Override).

## Процесс

1. **Собери список фич.** Разбери входные данные (список или путь к файлу).
2. **Выбери фреймворк.**
   - *RICE (по умолчанию):* Reach × Impact × Confidence / Effort.
   - *ICE:* Impact × Confidence × Ease.
   - *MoSCoW:* Must have, Should have, Could have, Won't have.
   - *WSJF (Weighted Shortest Job First):* `CoD / Job Size`, где `CoD` (Cost of Delay) = User Value + Time Criticality + Risk Reduction/Opportunity Enablement.
3. **Примени Regulatory Override (Регуляторный обход).** Выдели фичи, которые обязательны по закону, предписаниям ЦБ РФ или требованиям безопасности/комплаенса (152-ФЗ, 115-ФЗ). Эти фичи автоматически переносятся в категорию P0 (Must Have / Делаем сейчас) в обход любого математического скоринга, с пометкой «Регуляторное требование».
4. **Ранжируй фичи.** Отсортируй по убыванию оценки, с учетом P0-обхода.
5. **Проведи черту отсечения.** Распредели на: Делаем сейчас (этот спринт/квартал), Делаем следующим, Откладываем.
6. **Сохрани вывод** в текущей рабочей директории как `prioritized-backlog-[контекст].md`.

## Формат вывода

```
## Приоритизация бэклога: [Контекст]
- **Фреймворк:** [RICE / ICE / WSJF / MoSCoW]
- **Регуляторный комплаенс:** Включен (фичи с регуляторными требованиями имеют приоритет P0).

### 1. Таблица приоритизации
| Ранг | Фича | Метрики скоринга (Reach/Impact/CoD...) | Оценка / Категория | Статус | Причина приоритета / Примечание |
|------|------|----------------------------------------|--------------------|--------|---------------------------------|
| P0 (1) | [Фича 1] | — | **Regulatory Override** | Делаем сейчас | Требование 152-ФЗ по хранению персональных данных в РФ |
| 2 | [Фича 2] | [Reach=10k, Impact=2, Conf=80%, Eff=2] | 8000 (RICE) | Делаем сейчас | Высокий охват при малых усилиях |
| ... | ... | ... | ... | ... | ... |

### 2. Детализация по категориям (Черта отсечения)
- **Делаем сейчас (этот квартал):** [Список фич с рангом 1-N]
- **Делаем следующим:** [Список фич]
- **Откладываем:** [Список фич]

### 3. Обоснование оценок и компромиссов
- **Топ-приоритеты:** почему эти фичи оказались вверху бэклога.
- **Низкие приоритеты:** почему эти фичи отложены (например, высокая сложность при сомнительном эффекте).
- **Спорные моменты:** фичи с близким скором (разница < 10%), выбор между которыми требует дополнительных качественных аргументов.
```

## Правила

- Фичи, помеченные как Regulatory Override (комплаенс ЦБ РФ, законы РФ), должны идти на самый верх бэклога (P0/Must Have) без математического расчета. Безопасность и юридическая чистота бизнеса важнее продуктовых фич.
- Требуй единого масштаба оценок усилий (человеко-месяцы или Story Points) для всех фич в рамках одного расчета.
- Не допускай субъективного завышения Confidence. Если нет данных исследований или тестов — ставь Confidence = 50% и требуй сначала провести Discovery.
- Пиши на русском языке.

# СКИЛЛ: /okr-writer

# Написание OKR

Превратить расплывчатые бизнес-цели в чёткие, измеримые OKR. Никаких тщеславных метрик, заниженных целей, целей которые читаются как задачи.

## Процесс

1. **Разбери цель.** Пойми что бизнес пытается достичь и временной горизонт (по умолчанию: квартал).
2. **Напиши Objective.** Качественный, вдохновляющий, ограниченный по времени. Отвечает на «куда мы хотим прийти?» — а не «что мы будем делать?»
3. **Определи Key Results.** 3-5 измеримых результатов. Каждый должен иметь конкретную цифру. Отвечают на «как мы узнаем что добрались?»
4. **Свяжи с инициативами.** Какие активности будут двигать Key Results? Это фактические рабочие элементы.
5. **Проверь выравнивание.** Как этот OKR связан с компанией или вышестоящей командной целью?
6. **Определи анти-цели.** Что явно вне скоупа? Это предотвращает расширение скоупа и несогласованные усилия.

## Формат вывода

```
## OKR: [Квартал/Период]

### Objective
[Одно предложение. Качественный. Вдохновляющий. Чёткое направление. Ограниченный по времени.]

### Key Results
| # | Key Result | Базовое значение | Цель | Как измерять |
|---|-----------|-----------------|------|-------------|
| KR1 | [Конкретный измеримый результат] | [текущее] | [цель] | [источник данных] |
| KR2 | [Конкретный измеримый результат] | [текущее] | [цель] | [источник данных] |
| KR3 | [Конкретный измеримый результат] | [текущее] | [цель] | [источник данных] |
| KR4 | [Конкретный измеримый результат] | [текущее] | [цель] | [источник данных] |

### Инициативы
| Инициатива | Двигает KR | Ответственный | Усилия |
|-----------|-----------|--------------|--------|
| [Что будем делать] | KR1, KR2 | [команда/человек] | [S/M/L] |
| [Что будем делать] | KR2 | [команда/человек] | [S/M/L] |
| [Что будем делать] | KR3, KR4 | [команда/человек] | [S/M/L] |

### Выравнивание
- **Компанейская цель которую поддерживает:** [Какая вышестоящая цель]
- **Как связано:** [Одно предложение — причинно-следственная связь]

### Анти-цели
- [Чем этот Objective НЕ является]
- [Что мы явно НЕ будем делать в этом квартале]
- [Смежная область которую мы выбираем депривитизировать]
```

## Правила

- Objectives — это результаты, а не выпуски. «Запустить фичу X» — задача. «Стать предпочтительным инструментом для [персоны]» — Objective.
- Key Results должны иметь цифры. «Улучшить удержание» — не KR. «Увеличить 30-дневное удержание с 45% до 55%» — KR.
- Цели должны быть амбициозными, но не безумными. Хороший KR имеет примерно 70% шанс достижения. Если уверен на 100% — цель слишком низкая.
- Каждый KR должен указывать как будет измеряться и откуда берутся данные. Если не можешь измерить — это не KR.
- Максимум 3-5 Key Results. Более 5 означает что Objective слишком широкий — раздели его.
- Инициативы — НЕ Key Results. Инициативы — что ты делаешь. Key Results — что происходит как следствие.
- Анти-цели обязательны. Каждый OKR неявно депривитизирует что-то — назови это явно чтобы предотвратить дрейф.
- Если входная цель слишком расплывчата («вырасти бизнес»), спроси конкретику: какой сегмент, какая метрика, какой временной горизонт.
- Не пиши OKR для более чем одного Objective за раз если явно не просят. Фокус важнее широты.
- Пиши на русском языке.

# Документ о решении

Превратить неопределённость в структурированное решение. Одна страница, чёткие варианты, явные компромиссы и рекомендация которую можно защитить.

## Процесс

1. **Сформулируй решение.** Что именно мы решаем? Почему сейчас? Что произойдёт если ничего не делать?
2. **Определи ограничения.** Время, бюджет, технические ограничения, зависимости, организационные факторы.
3. **Сгенерируй варианты.** 2-4 реалистичных варианта. Всегда включай «ничего не делать» если это жизнеспособно. Каждый вариант должен быть реально другим, а не вариацией одной идеи.
4. **Оцени каждый вариант.** Плюсы, минусы, оценка усилий, уровень риска и обратимость.
5. **Дай рекомендацию.** Выбери один. Скажи почему. Явно укажи от чего отказываешься.
6. **Определи следующие шаги.** Если одобрено — что происходит в День 1?

## Формат вывода

```
## Решение: [Формулировка решения в одну строку]

### Контекст
[2-3 предложения. Почему это решение важно сейчас. Что его вызвало. Что произойдёт при задержке.]

### Ограничения
- [Ограничение 1]
- [Ограничение 2]
- [Ограничение 3]

### Варианты

#### Вариант A: [Название]
[1-2 предложения описания]
- **Плюсы:** [список]
- **Минусы:** [список]
- **Усилия:** [T-shirt размер + календарное время]
- **Риск:** [Низкий/Средний/Высокий — одно предложение почему]

#### Вариант B: [Название]
[1-2 предложения описания]
- **Плюсы:** [список]
- **Минусы:** [список]
- **Усилия:** [T-shirt размер + календарное время]
- **Риск:** [Низкий/Средний/Высокий — одно предложение почему]

#### Вариант C: [Название] (если применимо)
[та же структура]

### Сравнение
| Критерий | Вариант A | Вариант B | Вариант C |
|----------|-----------|-----------|-----------|
| Скорость получения ценности | [быстро/средне/медленно] | ... | ... |
| Влияние на пользователя | [высокое/среднее/низкое] | ... | ... |
| Усилия | [S/M/L/XL] | ... | ... |
| Риск | [низкий/средний/высокий] | ... | ... |
| Обратимость | [легко/сложно] | ... | ... |

### Рекомендация
**Выбираем Вариант [X].**
[2-3 предложения. Почему этот вариант побеждает. Какой конкретный фактор склоняет чашу весов.]

### От чего мы отказываемся
[Явно. У каждого выбора есть цена. Назови её.]

### Обратимость
[Можем ли мы изменить курс позже? Какой ценой? Когда это станет необратимым?]

### Следующие шаги (если одобрено)
1. [Действие] — [Ответственный] — [Когда]
2. [Действие] — [Ответственный] — [Когда]
3. [Действие] — [Ответственный] — [Когда]
```

## Правила

- Всегда давай рекомендацию. Документ о решении без рекомендации — просто меню. Займи позицию.
- «Ничего не делать» — валидный вариант. Включай его когда бездействие реально жизнеспособно, и честно назови его стоимость.
- Плюсы и минусы должны быть специфичны для этого решения, а не общими. «Быстрый выход на рынок» — плюс только если объяснишь быстрее чего и почему скорость важна здесь.
- Никогда не представляй более 4 вариантов. Если их больше — проблема недостаточно сужена.
- Обратимость обязательна. Необратимые решения заслуживают больше анализа. Обратимые — более быстрых действий.
- Раздел «От чего мы отказываемся» — не опциональный. Решения без признанных компромиссов — не настоящие решения.
- Максимум одна страница. Если документ длиннее 2 страниц — вероятно, это два решения. Раздели.
- Если входные данные слишком расплывчаты для генерации значимых вариантов — задай уточняющие вопросы. Не выдумывай контекст.
- Пиши на русском языке.

# Генератор релизных заметок

Превратить сырые артефакты разработки (git-коммиты, тикеты Jira, чейнджлоги, черновые заметки) в отполированные релизные заметки для пользователей. Пишите для людей, а не для инженеров.

## Процесс

1. Прочитай входные данные — либо из указанного файла, либо из вставленных в аргумент коммитов/тикетов.
2. Если указан путь к git-репозиторию, используй `git log --oneline` (в режиме чтения), чтобы извлечь последние коммиты.
3. Сгруппируй изменения по категориям: Новые фичи, Улучшения, Исправления багов.
4. Переведи каждое техническое изменение на понятный пользователю язык.
5. Определи тон из контекста. Если он не ясен, используй по умолчанию «Формальный B2B».
6. Напиши релизные заметки.
7. Сохрани результат в текущей рабочей директории как `release-notes-[версия-или-дата].md`.

## Формат вывода

### [Название релиза — описательное, а не просто номер версии]
**Версия:** [если применимо]
**Дата:** [дата релиза]

---

#### Новые фичи
Для каждой фичи:
- **[Название фичи]** — Что она делает и почему это важно для пользователя. Не то, как она была устроена внутри.

#### Улучшения
Для каждого улучшения:
- **[Что улучшилось]** — Что именно пользователь заметит в лучшую сторону. Будьте конкретны.

#### Исправления багов
Для каждого исправления:
- **[Что исправлено]** — Опишите проблему, с которой сталкивался пользователь, а не техническую первопричину.

---

#### Внутренние примечания (опционально, только по запросу)
- Технические детали, важные для поддержки, аккаунт-менеджеров или внутренних стейкхолдеров.
- Инструкции по миграции, фича-флаги, процент раскатки.
- Известные проблемы, которые не были исправлены в этом релизе.

## Гид по тональности

**Формальный B2B:**
- Профессиональный, чёткий, лаконичный.
- «Мы добавили...» / «Этот релиз содержит...»
- Подходит для корпоративных клиентов, официальных страниц чейнджлогов.

**Дружелюбный B2C:**
- Тёплый, разговорный, сфокусированный на выгоде.
- «Теперь вы можете...» / «Мы исправили ошибку, из-за которой...»
- Подходит для массовых продуктов, внутриаппных чейнджлогов.

**Внутренний:**
- Прямой, содержит технический контекст.
- Может ссылаться на номера тикетов, фича-флаги, названия команд.
- Подходит для апдейтов в Slack, внутренней базы знаний.

## Правила

- Пишите для ПОЛЬЗОВАТЕЛЕЙ, а не для инженеров. Переводите каждое изменение на язык пользы для человека.
- «Исправлен null pointer exception в сервисе авторизации» превращается в «Исправили проблему, из-за которой некоторые пользователи не могли войти в систему».
- «Рефакторинг модуля оплаты» становится Улучшением только если пользователи заметят разницу (например, скорость проведения платежа) — иначе опустите это.
- Внутренние рефакторинги без видимого эффекта для пользователя должны быть исключены из внешних заметок (при необходимости перенесите во Внутренние примечания).
- Группируйте связанные коммиты в один пункт релизных заметок. Пять коммитов по одной фиче = один буллит.
- Никогда не раскрывайте внутренние названия систем, коды ошибок или детали инфраструктуры в пользовательских разделах.
- Если тон не указан, используйте по умолчанию Формальный B2B.
- Каждый пункт должен отвечать на вопрос: «Почему пользователю должно быть не всё равно?»
- Пиши на русском языке.

# Технический переводчик

Прочитать технический документ и перевести его на язык продуктовых решений. Не упростить до примитива — а именно перевести. Цель — сохранить глубину и нюансы, сделав материал полезным для принятия решений по продукту.

## Процесс

1. Прочитай технический документ по указанному пути к файлу.
2. Определи ключевые технические концепции, решения и компромиссы.
3. Переведи каждый пункт на язык влияния на продукт и пользователей.
4. Выдели то, что PM обязан понять, отдельно от фонового контекста (nice-to-know).
5. Подсвети решения, которые PM должен принять или на которые должен повлиять.
6. Составь список вопросов, которые PM должен обсудить с разработкой.
7. Сохрани результат в текущей рабочей директории как `translated-[оригинальное-название-файла].md`.

## Формат вывода

### Суть (TL;DR)
1-2 предложения. О чём этот документ и почему это важно для продакт-менеджера?

### Что это значит для продукта
- Как это меняет то, что мы можем построить?
- Открывает ли это новые возможности или, наоборот, закрывает какие-то варианты?
- Влияет ли это на наш роадмап или сроки?
- Есть ли влияние на архитектуру продукта или платформенную стратегию?

### Что это значит для пользователей
- Заметят ли пользователи изменения? (Производительность, изменения в интерфейсе, новые фичи, ломающие изменения)
- Влияет ли это по-разному на разные сегменты пользователей?
- Потребуются ли миграции, будут ли простои или изменения в поведении систем, о которых пользователям важно знать?
- Есть ли влияние на данные пользователей, приватность или безопасность?

### Ключевые решения, которые должен принять PM
Для каждого решения:
- В чём суть решения?
- Какие есть варианты?
- Каковы компромиссы (в терминах продукта и бизнеса, а не технических)?
- Какова рекомендация команды разработки и почему?

### Вопросы к команде разработки
- Уточняющие вопросы, которые PM должен принести на следующую встречу.
- Сценарии формата «Что если...», которые не описаны в документе.
- Вопросы по срокам и ресурсам для валидации.

### Риски
- Технические риски, переведённые на язык последствий для продукта («Если миграция провалится, пользователи столкнутся с X»).
- Риски по срокам.
- Риски зависимостей.
- Допущения в документе, которые могут оказаться неверными.

### Глоссарий (при необходимости)
- Технические термины из документа с простыми определениями на человеческом языке.
- Включай только те термины, с которыми PM столкнется в будущем — пропускай разовый жаргон.

## Правила

- Не упрощай до примитива. Переводи. Продакт должен понимать инженерные нюансы, а не читать размытую версию.
- Чётко отделяй то, что PM ОБЯЗАН понять (влияет на решения), от справочного контекста.
- При описании компромиссов всегда формулируй их с точки зрения влияния на пользователей, сроки или стоимость.
- Если технический документ двусмысленен — подсвети эту двусмысленность, а не пытайся угадать смысл.
- Сохраняй ссылки на структуру исходного документа, чтобы PM мог легко вернуться к первоисточнику при необходимости.
- Если в документе есть код — не копируй его, а объясни, что он делает и почему это важно.
- Никогда не выдумывай продуктовые последствия, которые не подтверждаются техническим текстом.
- Пиши на русском языке.

# Генератор чеклиста запуска

Создать полный, специфичный для продукта чеклист запуска из описания продукта или фичи. Охватывает подготовку к запуску, день запуска и после запуска — с ответственными, сроками и блокирующими зависимостями.

## Процесс

1. Прочитай описание продукта или фичи.
2. Определи тип запуска (новый продукт, крупная фича, минорное обновление, бета) для калибровки глубины чеклиста.
3. Сгенерируй чеклист адаптированный под конкретный продукт — не шаблон.
4. Отметь блокирующие элементы против nice-to-have.
5. Включи тайминг (T-2 недели, T-1 неделя, День запуска, T+1 день, T+1 неделя).
6. Сохрани вывод в текущей рабочей директории как `launch-checklist-[название-продукта].md`.

## Формат вывода

### Обзор запуска
- **Продукт/Фича:** [название]
- **Целевая дата запуска:** [дата или TBD]
- **Тип запуска:** Новый продукт / Крупная фича / Минорное обновление / Бета
- **Целевая аудитория:** [конкретно]

---

### Подготовка к запуску (T-2 недели до T-1 день)

Формат каждого элемента: `- [ ] [Конкретный элемент] — **Ответственный:** [роль] — **Блокирует:** Да/Нет — **Срок:** [тайминг]`

#### Внутренняя готовность
Сгенерируй 4-6 элементов специфичных для данного продукта, охватывающих: приёмку от разработки, QA, нагрузочное тестирование (если применимо), стратегию выкатки и план отката. Включай только элементы релевантные для типа запуска.

#### Документация и поддержка
Сгенерируй 3-5 элементов специфичных для данного продукта, охватывающих: пользовательскую документацию, обучение команды поддержки и внутреннюю базу знаний. Адаптируй под модель поддержки продукта.

#### Подготовка коммуникаций
Сгенерируй 3-5 элементов специфичных для данного продукта, охватывающих: релизные заметки, анонсы (каналы зависят от продукта) и обучение сейлз/CS. Включай только каналы релевантные для аудитории данного продукта.

---

### День запуска (T-0)

#### Выкатка
Сгенерируй 3-5 элементов, охватывающих: деплой, верификацию, мониторинг и дежурство. Адаптируй под модель деплоя продукта (поэтапная выкатка, фича-флаги, полный запуск и т.д.).

#### Коммуникации
Сгенерируй 3-5 элементов, охватывающих: публикацию анонсов и оповещение внутренних команд. Включай только каналы релевантные для данного продукта.

#### Мониторинг
Сгенерируй 3-4 элемента, охватывающих: частоту ошибок, производительность, обращения пользователей и объём обращений в поддержку. Ссылайся на конкретные метрики из описания продукта.

---

### После запуска (T+1 день до T+2 недели)

#### Первые 24 часа
Сгенерируй 3-4 элемента, охватывающих: ревью мониторинга, триаж обращений в поддержку, первичную обратную связь и решение о расширении выкатки.

#### Первая неделя
Сгенерируй 4-5 элементов, охватывающих: отслеживание метрик против целей, ревью обратной связи, триаж багов, полную выкатку и ретроспективу.

#### Первые две недели
Сгенерируй 3-4 элемента, охватывающих: ревью метрик со стейкхолдерами, приоритизацию быстрых улучшений, обновление роадмапа и документирование запуска.

- Адаптируй глубину под тип запуска. Минорный баг-фикс не требует полного GTM чеклиста. Новый продукт — требует.

# Подготовка к встрече

Создать чёткий однострашничный бриф чтобы все вошли в синхроне и вышли с решениями. Встречи без подготовки — просто групповая терапия.

## Процесс

1. **Разбери контекст встречи.** Пойми тему, кто вероятно будет присутствовать и что инициировало встречу.
2. **Определи цель.** С чем группа должна выйти? Решение, выравнивание, план или переданная информация.
3. **Сгенерируй ключевые вопросы.** 3-5 конкретных вопросов, на которые должна ответить встреча. Не открытые («мысли по X?»), а чёткие («выбираем A или B?»).
4. **Определи предварительный материал.** Что участникам нужно знать до встречи чтобы не тратить время на введение в контекст.
5. **Перечисли ожидаемые решения.** Какие выборы будут сделаны на этой встрече и у кого финальное слово.
6. **Установи паркинг.** Какие темы смежные, но явно вне скоупа данной встречи.

## Формат вывода

```
## Бриф встречи: [Тема]
**Дата:** [если указана] | **Продолжительность:** [если указана] | **Тип:** [Решение / Выравнивание / Брейнсторм / Ревью]

### Контекст
[1-2 предложения. Почему эта встреча происходит сейчас. Что её инициировало.]

### Цель
Выйти со встречи с: [конкретный результат — решение по X, выравнивание по Y, план по Z]

### Ключевые вопросы для ответа
1. [Конкретный, отвечаемый вопрос]
2. [Конкретный, отвечаемый вопрос]
3. [Конкретный, отвечаемый вопрос]
4. [Конкретный, отвечаемый вопрос] (если нужно)
5. [Конкретный, отвечаемый вопрос] (если нужно)

### Предварительный материал
Участники должны ознакомиться до встречи:
- [Документ/данные/контекст + почему это важно]
- [Документ/данные/контекст + почему это важно]
- [Ключевая цифра или факт, который нужно иметь в голове]

### Ожидаемые решения
| Решение | Варианты | Принимающий решение |
|---------|----------|---------------------|
| [Что нужно решить] | [A vs. B vs. C] | [У кого финальное слово] |
| [Что нужно решить] | [A vs. B] | [У кого финальное слово] |

### Паркинг (НЕ обсуждать)
- [Тема которая уведёт встречу в сторону]
- [Смежная но отдельная проблема — запланировать отдельную встречу]
- [Интересно, но не срочно]

### Предлагаемая повестка (если указано время)
| Время | Тема | Ведущий |
|-------|------|---------|
| [5 мин] | Контекст + цель | [фасилитатор] |
| [15 мин] | [Ключевой вопрос 1-2] | [ответственный человек] |
| [15 мин] | [Ключевой вопрос 3-4] | [ответственный человек] |
| [10 мин] | Решение + следующие шаги | [фасилитатор] |
| [5 мин] | Итоги + action items | [фасилитатор] |
```

## Правила

- Цель должна быть конкретной. «Обсудить роадмап» — не цель. «Решить приоритеты Q3 и назначить ответственных» — цель.
- Ключевые вопросы должны быть отвечаемыми на встрече. Если вопрос требует исследования которое не проведено — он относится к предварительному материалу как action item, а не к встрече.
- Каждая встреча должна иметь хотя бы одно ожидаемое решение. Если нет решений — поставь под сомнение нужна ли встреча вообще или лучше email. Скажи это явно.
- Паркинг — не опциональный. У каждой встречи есть риск расширения скоупа. Назови вероятные отступления заранее.
- Предварительный материал должен быть конкретным. «Прочитать документ» — бесполезно. «Прочитать разделы 2-3 PRD, сфокусироваться на таблице компромиссов» — полезно.
- Весь бриф — одна страница. Если длиннее — скоуп встречи слишком широкий, предложи разбить.
- Если входные данные — только название встречи без контекста, задай 2-3 уточняющих вопроса вместо угадывания.
- Предлагай распределение времени только если продолжительность встречи указана.
- Пиши на русском языке.

# Анализатор метрик (Metrics Analyzer)

Превратить сырые данные о поведении пользователей или транзакциях в понятные инсайты. Скилл поддерживает стандартный анализ данных (тренды, сегменты, аномалии), а также специализированный финтех/банковский режим для работы с кредитным портфелем, фрод-активностью и сезонными колебаниями спроса.

## Процесс

1. **Загрузи и разбери данные.** (Прочитай CSV-файл или разбери текст с помощью Python/Pandas).
2. **Включи финтех-фильтры (при наличии финансовых данных):**
   - *Сезонность (Seasonality):* учитывай классические сезонные колебания (например, падение транзакций в начале января, пик выдачи кредитов и ипотеки в декабре перед закрытием года, сезонные просадки летом).
   - *Фрод-всплески (Fraud Spikes):* ищи аномальные пики транзакций с высоким процентом отказов (chargeback) или возвратов, резкий рост мелких транзакций (признак брутфорса/кардинга) или нетипичные гео-переходы.
   - *Группировка кредитных когорт (Vintage Cohorts):* группируй кредиты/карты по месяцу выдачи (выдачи января, выдачи февраля) для анализа накопленной просрочки (NPL) и окупаемости.
3. **Обнаружь общие тренды и аномалии.** Рассчитай отклонения более чем на 2 стандартных отклонения от среднего значения.
4. **Сформулируй 3 actionable инсайта** с рекомендациями для бизнеса и продукта.
5. **Сохрани вывод** в текущей рабочей директории как `metrics-analysis-[дата].md`.

## Формат вывода

```
## Анализ метрик: [Контекст данных]
- **Диапазон данных:** [Дата старта] по [Дата окончания]
- **Специфика анализа:** [Стандартный / Финтех-анализ]

### 1. Ключевые показатели и Тренды
| Метрика | Текущее значение | Предыдущий период | Отклонение (MoM / YoY) | Оценка тренда |
|---------|------------------|-------------------|------------------------|---------------|
| [метрика] | [X] | [Y] | [+/-Z%] | [Рост / Падение / Стабильно] |

### 2. Аномалии и Фрод-мониторинг
- **Детекция аномалий:** [описание всплесков/падений, выходящих за границы 2-х стандартных отклонений].
- **Фрод-сигналы (для транзакционных данных):** [обнаружены ли всплески возвратов, дублирующиеся платежи, аномалии по картам/IP].

### 3. Сезонный анализ и Когортные сдвиги
- **Влияние сезонности:** [оценка влияния времени года/праздников на метрики; например, «Текущий спад выдач на 15% в январе соответствует исторической сезонности праздничных дней, паники нет»].
- **Винтажный анализ (Vintage / Cohort):** [как ведут себя когорты по дате выдачи кредитов; качество портфеля на Month 3, Month 6].

### 4. Топ-3 Actionable инсайта
1. **[Заголовок инсайта]:** [Что показывают данные] — [Почему это важно] — [Предлагаемое действие для команды продукта]
2. **[Заголовок инсайта]:** [Что показывают данные] — [Почему это важно] — [Предлагаемое действие]
3. **[Заголовок инсайта]:** [Что показывают данные] — [Почему это важно] — [Предлагаемое действие]
```

## Правила

- Не делай выводов на глаз. Всегда запускай скрипты на Python (Pandas/NumPy) для подсчета средних, медиан, стандартных отклонений и изменений MoM/YoY.
- Всегда разделяй реальное ухудшение метрик и сезонные колебания. Уменьшение объема транзакций в первой половине января — это нормальная сезонность ритейла и банкинга, а не признак сбоя продукта.
- Финтех-инсайты должны содержать защитные рекомендации. Нельзя рекомендовать «упростить скоринг для роста конверсии», не указав риск роста просрочки (NPL) и маржинальных потерь.
- Пиши на русском языке.

# Фасилитатор ретроспектив

Два режима: ГЕНЕРАЦИЯ доски ретроспектив с подсказками или СИНТЕЗ сырых заметок с ретро в темы и конкретные действия (экшены). Ретроспективы без последующих действий — это просто сессии жалоб.

## Процесс

### Определение режима
- Если на входе название формата (4Ls, start-stop-continue, sailboat) → режим ГЕНЕРАЦИИ.
- Если на входе вставленные заметки, буллиты или сырой фидбек → режим СИНТЕЗА.
- Если неясно — спроси.

### Режим ГЕНЕРАЦИИ
1. **Выбери формат.** Определи, какой формат ретроспектив нужен пользователю.
2. **Создай разделы.** Построй структуру доски для этого формата с описаниями разделов.
3. **Добавь стартовые вопросы.** 2-3 наводящих вопроса на раздел, чтобы запустить обсуждение.
4. **Добавь советы по фасилитации.** Как эффективно провести каждый раздел.
5. **Предложи тайм-план.** По умолчанию на основе 60-минутной встречи.

### Режим СИНТЕЗА
1. **Собери заметки.** Прочитай все сырые входные данные — стикеры, списки, свободный текст.
2. **Сгруппируй по темам.** Объедини связанные пункты. Назови каждую тему понятно и ёмко.
3. **Приоритезируй темы.** Ранжируй по частоте (сколько людей упомянули) и влиянию (как сильно влияет на команду).
4. **Сформируй экшены (action items).** У каждого действия должно быть: что сделать, кто делает и к какому сроку.
5. **Выяви паттерны.** Если доступен контекст прошлых ретро — отметь повторяющиеся темы.

## Поддерживаемые форматы

### 4Ls
- **Loved (Что понравилось):** Что шло отлично и что мы хотим продолжать делать.
- **Learned (Чему научились):** Что мы открыли для себя или какие инсайты получили.
- **Lacked (Чего не хватало):** Чего не хватало или было недостаточно.
- **Longed For (Чего хотелось бы):** Что мы очень хотели бы иметь на будущее.

### Start-Stop-Continue
- **Start (Начать):** Что нам стоит начать делать.
- **Stop (Прекратить):** Что нам нужно перестать делать.
- **Continue (Продолжать):** Что отлично работает и это надо закрепить.

### Sailboat (Парусник)
- **Ветер (что двигает нас):** Силы, толкающие нас вперёд.
- **Якорь (что тормозит нас):** Что замедляет наш прогресс.
- **Рифы (риски впереди):** Препятствия, которые могут нам навредить.
- **Остров (цель):** Куда мы стремимся добраться.

## Правила

- В режиме ГЕНЕРАЦИИ стартовые вопросы должны быть адаптированы под контекст команды, если он предоставлен. Шаблонные вопросы типа «Что шло хорошо?» впустую тратят время людей.
- В режиме СИНТЕЗА у каждой темы должен быть хотя бы один экшен. Тема без действия — просто констатация факта.
- Экшены должны быть конкретными и иметь владельца. «Улучшить коммуникацию» — не экшен. «Запланировать ежедневный 15-минутный синк на следующие 2 недели — владелец [Имя]» — экшен.
- Не выделяйте более 5 тем при синтезе. Если их больше — объедините мелкие.
- Максимум 5 экшенов на выходе. Команды, уходящие с ретро с 10 задачами, выполняют ноль.
- Всегда выводите сигнал здоровья команды в режиме синтеза. Ретроспекция нужна для непрерывных улучшений — отслеживайте вектор.
- Если заметки слишком скудные для синтеза (менее 5 пунктов) — скажите об этом прямо и укажите, что команде необходимо развивать психологическую безопасность для честного фидбека.
- Пиши на русском языке.

# Апдейт для стейкхолдеров

Превратить разрозненные заметки в чёткое обновление для стейкхолдеров. Руководители просматривают бегло — каждое предложение должно заслуживать своего места.

## Процесс

1. **Разбери сырые заметки.** Прочитай входные данные — заметки встреч, маркированные списки, вставленные Slack-треды, статус-апдейты — всё что предоставил пользователь.
2. **Определи формат.** По умолчанию — email. Пользователь может запросить: email, Slack-сообщение или тезисы для презентации. Адаптируй тон и структуру соответственно.
3. **Извлеки ключевую информацию.** Категоризируй всё в: прогресс (что выпущено/продвинулось), блокеры (где застряли/нужна помощь), необходимые решения (конкретные запросы) и предстоящие приоритеты.
4. **Напиши TL;DR.** Максимум два предложения. Самое важное и единственное что нужно от читателя.
5. **Структурируй апдейт.** Следуй формату вывода для выбранного носителя.
6. **Квантифицируй.** Заменяй расплывчатые формулировки конкретикой. «Хороший прогресс» становится «выпустили 3 из 5 запланированных фич».

## Формат вывода

### Формат Email (по умолчанию)
```

**TL;DR:** [Два предложения. Что произошло + что нужно.]

**Прогресс**
- [Что выпущено или продвинулось — конкретно]
- [Что выпущено или продвинулось]
- [Что выпущено или продвинулось]

**Блокеры**
- [Что застряло] — Нужна [конкретная помощь] от [конкретного человека/команды]
- [Что застряло] — Нужна [конкретная помощь]

# Дизайн A/B-теста и ML-экспериментов (ab-test-design)

Спроектировать план продуктового эксперимента или тестирования ML-модели. Скилл помогает продакту поставить гипотезу, выбрать первичную и защитную метрики, рассчитать размер выборки и длительность, а также спроектировать логику теневого тестирования (shadow testing) и раскатки Champion/Challenger для алгоритмов искусственного интеллекта.

## Процесс

1. **Уточни тип эксперимента.**
   - *Классический A/B-тест:* сравнение интерфейсов или логики на пользователях.
   - *ML-эксперимент:* сравнение моделей рекомендаций, ранжирования или скоринга.
2. **Сформулируй гипотезу.** Используй шаблон: `Если мы [изменение/модель], то [метрика] [направление] на [величина] потому что [причинно-следственная связь]`.
3. **Определи метрики.** Выбери одну первичную метрику (решение о победе), 2-3 вторичные и 2-3 защитные (метрики стабильности, крашей, ошибок ML).
4. **Спроектируй ML-варианты (для ML-тестов):**
   - *Shadow Testing (Теневой тест):* запуск новой модели на проде в режиме «записи» — она считает скоры/рекомендации в фоне, но пользователю показывается старая логика. Оценка оффлайн-метрик на реальном потоке данных.
   - *Champion/Challenger:* раскат трафика на старую базовую модель (Champion) и новые экспериментальные модели (Challengers) в реальном времени.
5. **Оцени размер выборки и длительность.** На основе MDE (Minimum Detectable Effect), базовой конверсии и дневного трафика.
6. **Сохрани вывод** в текущей рабочей директории как `ab-test-design-[контекст].md`.

## Формат вывода

```
## Дизайн эксперимента: [Название]
- **Тип теста:** [Классический A/B / ML-эксперимент]

### 1. Гипотеза
Если мы [изменение / внедрим модель X], то [первичная метрика] [направление] на [X%] потому что [причина].

### 2. Метрики теста
- **Первичная метрика:** [одна главная метрика, например: Ср. чек по выдаче ипотеки]
- **Вторичные метрики:** [конверсия в сделку, утилизация лимита]
- **Защитные метрики (Guardrail):** [для ML: latency модели p95 < 200мс, доля ошибок модели 5xx < 0.1%, уровень просрочки NPL < 2.5%]

### 3. Схема раската и Варианты (Rollout Strategy)
- **Контроль (Champion):** [текущее состояние / текущая модель ранжирования]
- **Тест (Challenger):** [новое решение / новая ML-модель]
- **Режим тестирования:**
  - *Shadow Testing (если применимо):* [описание теневого сбора данных: пишем логи новой модели 7 дней, оцениваем оффлайн Precision/Recall на реальных заявках без влияния на клиентов].
  - *Сплит-тест:* [доля распределения трафика, например, 90/10 для новой сложной ML-модели на старте].

### 4. Размер выборки и Длительность
- **Минимально обнаруживаемый эффект (MDE):** [X%]
- **Базовая конверсия/значение:** [Y%]
- **Размер выборки на вариант:** [N пользователей/заявок]
- **Продолжительность теста:** [X дней / недель] при дневном трафике [Y].

### 5. Риски и критерии досрочной остановки
- **Критерий экстренного отключения (Kill Switch):** [например, падение одобряемости кредитов более чем на 5% или рост ошибок модели].
- **Проблема подглядывания (P-peeking):** запрет на принятие решений до набора полной выборки.
```

## Правила

- Только одна первичная метрика для принятия решения. Попытка оптимизировать 5 метрик одновременно ведет к ложноположительным результатам.
- Для ML-моделей защитные метрики производительности (latency, error rate) обязательны. Тяжелая модель с высокой точностью, но увеличивающая время ответа (latency) на 1 секунду, убьет конверсию в покупку.
- Если MDE слишком мал, а трафик низкий (тест будет идти > 4 недель), требуй пересмотра дизайна (увеличить MDE, выбрать прокси-метрику с более высокой частотой событий).
- Пиши на русском языке.

# Бизнес-модель маркетплейса (Marketplace Model)

Создать структурированный анализ бизнес-модели двустороннего маркетплейса. Этот скилл помогает продакту спроектировать сбалансированную экономику платформы, решить проблему холодного старта спроса и предложения, рассчитать оптимальные комиссии и определить ключевые метрики ликвидности.

## Процесс

1. **Определи стороны платформы.** Кто поставляет ценность (Supply / Продавцы, исполнители) и кто её потребляет (Demand / Покупатели, заказчики)?
2. **Определи тип маркетплейса.** (Транзакционный — WB/Ozon, лидогенерационный — Профи/Авито, шеринг — Airbnb, SaaS-enabled — когда одна из сторон использует софт платформы).
3. **Разработай стратегию холодного старта (Курица и Яйцо).** Как привлечь первую сторону без наличия второй? (Примеры: субсидирование одной стороны, создание ценности "Single-Player Mode" через полезный софт, агрегация открытых данных о предложении).
4. **Спроектируй модель монетизации.** Какова комиссия платформы (take rate)? Кто её платит — покупатель, продавец или оба? Есть ли дополнительные источники дохода (подписки, платное продвижение карточек, логистические услуги)?
5. **Опиши метрики ликвидности.** Как измерить, что рынок «живой»? (Метрики: доля заполненных заказов, время до первого отклика, ликвидность продавцов).
6. **Сохрани вывод** в текущей рабочей директории как `marketplace-model-[название-продукта].md`.

## Формат вывода

```
## Бизнес-модель маркетплейса: [Название]

### 1. Архитектура сторон (Demand & Supply)
- **Сторона спроса (Demand / Покупатели):** [кто они, какова частота покупки, средний чек, ключевая мотивация прийти на платформу]
- **Сторона предложения (Supply / Продавцы):** [кто они, их объем товаров/услуг, профессиональный уровень, барьер входа на платформу]
- **Тип маркетплейса:** [Транзакционный / Classifieds / On-demand / SaaS-enabled]

### 2. Стратегия решения проблемы «Курицы и Яйца» (Холодный старт)
- **Какую сторону привлекаем первой:** [спрос или предложение] и почему.
- **Тактика холодного старта:**
  - *Субсидирование:* [например, даем бесплатные лиды специалистам на старте или гарантируем минимальный доход таксистам]
  - *Инструмент для одной стороны (Single-player tool):* [например, CRM-система для салонов красоты, которая работает без клиентов с маркетплейса]
  - *Создание искусственного предложения:* [парсинг объявлений, ручной онбординг партнеров]

### 3. Монетизация и Take Rate
- **Базовый Take Rate:** % комиссии с транзакции. Кто платит комиссию (например, 15% с продавца, 0% с покупателя). Обоснование размера комиссии.
- **Дополнительные потоки (Value-Added Services):**
  - Продвижение предложений (рекламная модель, платное поднятие объявлений).
  - Логистические/финансовые услуги (доставка силами платформы, страхование сделок, эквайринг).
  - SaaS-подписка для продавцов (CRM, расширенная аналитика).

### 4. Метрики ликвидности рынка (Marketplace Liquidity)
- **Buyer Liquidity (Ликвидность для покупателя):** % поисковых запросов или визитов, завершившихся успешной покупкой (Search-to-Buy Conversion).
- **Seller Liquidity (Ликвидность для продавца):** % активных карточек товаров или исполнителей, совершивших хотя бы одну продажу за период времени (активный ассортимент).
- **Match Rate / Fill Rate:** % заявок на услуги, на которые откликнулся хотя бы один исполнитель, или время до первого отклика.

### 5. Защита платформы от обхода (Disintermediation / Leakage)
- **Как пользователи могут обойти платформу:** [например, договориться о регулярных услугах напрямую без комиссии маркетплейса]
- **Меры борьбы с обходом:**
  - Удобство и безопасность (безопасная сделка, эскроу, страховка).
  - Накопительная ценность (рейтинг исполнителя, история заказов, которая дает ему новые заказы).
  - Штрафные санкции (блокировка аккаунта при передаче контактов в чате).

### 6. Ключевые риски бизнес-модели
- **Риск дисбаланса:** избыток продавцов при отсутствии покупателей (и наоборот). Как балансируем маркетинг.
- **Качество предложения (Supply Quality):** как защищаем покупателя от некачественного товара/услуги (рейтинги, отзывы, модерация).
```

## Правила

- Не позволяй продакту игнорировать проблему обхода платформы (leakage). Если исполнитель и заказчик могут легко уйти в офлайн после первой сделки — маркетплейс потеряет бизнес-модель. Продакт должен явно спроектировать ценность нахождения внутри платформы для повторных сделок.
- Оспаривай абстрактные показатели ликвидности. «У нас много пользователей» — не метрика ликвидности. Метрикой является конверсия визита в покупку или время до получения услуги.
- Take Rate должен быть экономически обоснован для продавцов. Слишком высокая комиссия вытолкнет их к конкурентам или в офлайн. Слишком низкая — не покроет CAC платформы.
- Пиши на русском языке.

# Алгоритмы матчинга в On-demand сервисах

Создать продуктовую спецификацию требований для алгоритма матчинга (сопоставления спроса и предложения) в сервисах реального времени (такси, экспресс-доставка, клининг, курьеры). Помогает продакту настроить автоматическое распределение заказов, спроектировать динамическое ценообразование (Surge Pricing) для балансировки рынка в пиковые часы и минимизировать время ожидания для пользователей.

## Процесс

1. **Определи модель сопоставления.** Как происходит матчинг: мгновенно ближайшему (Greedy match), пакетами каждые N секунд (Batch match), или через аукцион среди исполнителей?
2. **Спроектируй гео-зонирование (Geohash / Hexagons).** Как платформа делит карту на зоны для оценки баланса спроса и предложения? (Использование сетки гексагонов Uber H3 для локального анализа).
3. **Разработай динамическое ценообразование (Surge Pricing).** Каковы триггеры включения наценки? (Если количество свободных исполнителей в зоне < X при росте заказов > Y, цена умножается на коэффициент).
4. **Опиши логику диспетчеризации (Dispatching Logic).** Какие факторы влияют на приоритет назначения? (Расстояние до клиента, рейтинг исполнителя, время нахождения на смене, тип транспорта).
5. **Определи метрики эффективности матчинга.** Match Rate (% успешных сопоставлений), ETA (время подачи исполнителя), Cancellation Rate (отмены заказов из-за долгого поиска), средний доход исполнителя в час (Earnings per Hour).
6. **Сохрани вывод** в текущей рабочей директории как `matching-spec-[название-продукта].md`.

## Формат вывода

```
## Спецификация алгоритма матчинга: [Название]

### 1. Архитектура сопоставления (Matching Engine)
- **Модель матчинга:** [Batch matching (пакетный матчинг каждые 3-5 секунд для минимизации суммарного расстояния) / Greedy matching (назначение первому свободному)]
- **Входные параметры алгоритма:** [координаты клиента, координаты и направление движения свободных исполнителей, их рейтинг, класс обслуживания, ETA]

### 2. Динамическое ценообразование (Surge Pricing & Balance)
- **Сетка гео-анализа:** деление карты на гексагоны (индексы Uber H3, уровень детализации разрешения, например, H3 Resolution 8).
- **Алгоритм расчета коэффициента (Surge Multiplier):**
  - Оценка спроса: количество открытых приложений/попыток заказа в гексагоне за последние 2 минуты.
  - Оценка предложения: количество свободных исполнителей в этом и соседних гексагонах.
  - Расчет коэффициента: формула пересчета базовой стоимости на мультипликатор (например, от x1.0 до x3.5) для снижения спроса и привлечения исполнителей из соседних зон.

### 3. Алгоритм диспетчеризации (Dispatching & Queues)
- **Приоритизация назначения:**
  - `Скор исполнителя = Вес расстояния (ETA) × Вес рейтинга × Коэффициент типа транспорта`.
- **Логика очередей (Dispatching Radius):** постепенное расширение радиуса поиска исполнителя (например, 1 км -> 2 км -> 3 км) при отсутствии откликов.
- **Предотвращение холостого пробега:** алгоритмы назначения заказа в процессе завершения текущей поездки (Back-to-back orders).

### 4. Поведенческие стимулы сторон (Marketplace Incentives)
- **Для исполнителей:** доплаты за работу в «горячих» гексагонах (Surge zones), бонусы за выполнение целей по количеству заказов в сутки.
- **Для клиентов:** предложение отложить заказ на 5-10 минут (когда спадет Surge), переключиться на другой тариф/класс обслуживания.

### 5. Метрики качества матчинга (On-demand KPIs)
- **Match Rate / Fill Rate:** % заказов, на которые успешно назначен исполнитель.
- **Cancellation Rate (Спрос):** % отмен заказов клиентами до назначения исполнителя (из-за высокой цены или долгого поиска).
- **Driver Acceptance Rate:** % предложенных заказов, которые исполнители приняли (низкий % говорит о невыгодных тарифах или плохих маршрутах).
- **Pickup ETA:** среднее время подачи курьера/водителя к клиенту после назначения.
```

## Правила

- Избегай Greedy-матчинга (назначение ближайшему в данную секунду) на высоконагруженных рынках. Пакетный матчинг (Batching) снижает среднее расстояние подачи по системе на 15-20% по сравнению с Greedy.
- Surge Pricing обязателен. Без динамической наценки в дождь или час-пик маркетплейс столкнется с 100% нехваткой исполнителей, а пользователи — с бесконечным поиском заказа (разрушение UX).
- Проектируй защиту от накруток Surge. Исполнители не должны иметь возможность сговариваться и выходить из сети одновременно, чтобы искусственно поднимать коэффициенты в зоне.
- Пиши на русском языке.

- **API для продавцов:** необходимость интеграции с учетными системами продавцов (1С, МойСклад).
- **Инструменты продвижения:** рекламные кампании, участие в общих промо-акциях платформы, бустинг в поиске.

### 5. Риски платформы и Защита от фрода
- **Товарный фрод:** продажа реплик, контрафакта, некачественного товара (система модерации брендов, контрольные закупки).
- **Финансовый фрод:** накрутка отзывов и самовыкупы (детекция аномальной активности, защита от накруток рейтинга).
- **Сговор:** мошенничество продавцов с покупателями (защита сделки через эскроу-механики платформы).
```

## Правила

- Помни про «проблему курицы и яйца» на маркетплейсах. Продавцы не придут, если нет покупателей, а покупатели — если нет ассортимента. Спецификация должна отвечать на вопрос: как этот путь продавца помогает сбалансировать ликвидность маркетплейса на ранних этапах.

# Поиск и ранжирование (Search & Ranking)

Создать детальную спецификацию требований к поисковой выдаче и алгоритмам ранжирования для каталогов, маркетплейсов или classifieds (типа WB, Ozon, Авито, ЦИАН). Помогает продакту спроектировать сбалансированную поисковую логику, учитывающую текстовое соответствие, коммерческие интересы платформы и поведенческие метрики.

## Процесс

1. **Определи этапы работы поиска.** Парсинг запроса (синонимы, исправление опечаток) → Извлечение кандидатов (фильтрация) → Ранжирование (сортировка кандидатов).
2. **Определи группы факторов ранжирования.**
   - *Текстовая релевантность:* совпадение с заголовком, описанием, характеристиками.
   - *Поведенческие факторы:* CTR карточки в выдаче, конверсия в корзину/покупку, процент отказов.
   - *Коммерческие/Бизнес факторы:* маржинальность товара, стоимость доставки, платный бустинг (реклама), рейтинг продавца.
3. **Разреши конфликт интересов (Релевантность vs. Выручка).** Как сбалансировать выдачу, чтобы пользователь находил именно то, что ищет (релевантность), но платформа зарабатывала (приоритет товаров с высокой комиссией или платной рекламой).
4. **Реши проблему «Холодного старта» (Cold Start).** Как новые товары/карточки получают первые показы и просмотры, чтобы алгоритм мог оценить их конверсию (бустинг новинок).
5. **Определи метрики качества поиска.** NDCG (Normalized Discounted Cumulative Gain), CTR выдачи, доля пустых поисковых выдач (Zero Results Rate), конверсия из поиска в покупку.
6. **Сохрани вывод** в текущей рабочей директории как `search-spec-[контекст].md`.

## Формат вывода

```
## Спецификация поиска и ранжирования: [Название продукта]

### 1. Архитектура поиска и Обработка запросов
- **Тип каталога:** [например, маркетплейс товаров / аренда недвижимости]
- **Обработка текстового ввода (NLP/Search Engine):**
  - Распознавание опечаток и автоисправление.
  - Поддержка синонимов (например, «мобильник» = «смартфон»).
  - Автодополнение (Search Autocomplete) с подсказками популярных категорий.

### 2. Формула ранжирования (Ranking Factors)
Факторы, определяющие позицию карточки в выдаче:

| Группа факторов | Конкретные параметры | Вес в формуле (Высокий/Средний/Низкий) | Обоснование |
|-----------------|----------------------|---------------------------------------|-------------|
| **1. Текстовые** | Совпадение в названии, тегах, описании | [Высокий] | Базовый гигиенический фактор релевантности |
| **2. Поведенческие** | CTR карточки, конверсия в корзину, выкуп | [Высокий] | Алгоритм должен поднимать то, что реально покупают |
| **3. Коммерческие** | Размер комиссии (take rate), скорость доставки | [Средний] | Банку/Платформе выгодно везти быстрее и зарабатывать больше |
| **4. Рекламные** | Платное продвижение продавцом | [Средний] | Спонсируемые карточки поднимаются на фиксированные позиции |

### 3. Балансировка выдачи (Relevance vs. Monetization)
- **Правила рекламного бустинга:** на каких позициях выводятся спонсируемые товары (например, 1-я, 5-я, 9-я позиции в выдаче).
- **Пессимизация (Downranking):** за что карточки опускаются в выдаче (низкий рейтинг продавца, высокий процент возвратов товара, долгая доставка, отсутствие товара на складе).

### 4. Решение проблемы «Холодного старта» (Cold Start Strategy)
- **Бустинг новинок:** новые карточки без статистики продаж автоматически получают временный подъем (например, +10% к скору на первые 500 показов).
- **Сбор поведенческих сигналов:** как только карточка набирает тестовые показы, бустинг отключается, и её позиция определяется реальной конверсией.

### 5. Метрики качества поиска (Search KPIs)
- **Zero Results Rate (Доля пустых выдач):** % поисковых запросов, которые вернули 0 результатов (цель < 1-2%).
- **NDCG (Normalized Discounted Cumulative Gain):** мера релевантности выдачи (насколько лучшие результаты находятся в самом верху).
- **Search Conversion (Конверсия поиска):** конверсия из клика по результату поиска в целевое действие (покупка/заявка).
- **Доля переходов на вторую страницу:** высокий показатель говорит о том, что первая страница выдачи нерелевантна.
```

## Правила

- Запрещай жесткое ранжирование только по дате добавления или только по цене, если это не выбрано пользователем в фильтрах. Выдача по умолчанию (умная сортировка) должна строиться на комбинации факторов релевантности и конверсии.
- Метрика Zero Results Rate обязательна. Пустая выдача — это тупик для пользователя. Спецификация должна описывать fallback-сценарии (поиск по похожим словам, рекомендации категорий).
- Не допускай доминирования рекламных карточек в топе выдачи. Если первые 10 товаров будут только рекламными и нерелевантными запросу — пользователь уйдет с платформы.
- Пиши на русском языке.

# Бизнес-модель классифайдов (Classifieds Model)

Создать детальную спецификацию требований для продукта в сфере досок объявлений (Classifieds) и маркетплейсов услуг (аналоги Авито, ЦИАН, Профи.ру, Яндекс Услуги). Скилл помогает продакту спроектировать воронку размещения объявлений, выбрать модель монетизации (платные размещения, комиссии за сделку, оплата за контакты), оценить качество контента и настроить метрики ликвидности.

## Процесс

1. **Определи вертикаль классифайда.** (Недвижимость, авто, работа, товары, услуги). Какова специфика сделки (высокий чек/низкая частота vs. низкий чек/высокая частота)?
2. **Спроектируй модель монетизации листингов.**
   - *Freemium лимит:* сколько объявлений пользователь может разместить бесплатно в месяц в данной категории.
   - *Платные услуги (VAS - Value Added Services):* бустинг (поднятие в топ, увеличение просмотров в X раз), выделение цветом, VIP-размещение.
   - *Pay-per-Lead / Pay-per-Contact:* оплата за просмотр номера телефона или начало чата (модель Профи.ру / Авито Услуги).
3. **Разработай критерии качества объявления (Listing Quality Score).** Как алгоритм оценивает полноту описания, качество фотографий, адекватность цены и рейтинг продавца для принятия решения о модерации.
4. **Спроектируй защиту от дублей и фрода.** Как выявлять повторные объявления (спам), фейковые объекты (например, несуществующие квартиры-«фонари» в недвижимости), мошеннические схемы с предоплатой.
5. **Определи метрики ликвидности классифайда.** Reply Rate (доля объявлений с откликами), Time-to-Reply (скорость ответа продавца), конверсия из просмотра в контакт (View-to-Contact).
6. **Сохрани вывод** в текущей рабочей директории как `classifieds-spec-[название-продукта].md`.

## Формат вывода

```

### 1. Вертикаль и Целевая аудитория
- **Специализация:** [например, классифайд аренды коммерческой недвижимости]
- **Профиль участников:**
  - *Подающие объявления (Supply):* [частные лица / профессиональные риелторы / агентства]
  - *Ищущие (Demand):* [бизнес, предприниматели]
- **Сложность сделки:** [низкочастотная сделка с длинным циклом принятия решения]

### 2. Модель монетизации и Тарификация
- **Лимиты бесплатного размещения:** [например, 1 бесплатное объявление в категории «Офисы» раз в 90 дней, далее — платный листинг].
- **Пакетные подписки (для B2B):** тарифы для агентств (например, пакет на 50 активных объявлений в месяц + персональная страница).
- **Value-Added Services (VAS-пакеты):**
  - Бустинг: увеличение охвата в 2/5/10 раз на 1 или 7 дней (алгоритм динамически рассчитывает цену продвижения в зависимости от конкуренции в категории и гео).
  - Премиум-оформление: увеличенная карточка в ленте, подсвеченный заголовок.

### 3. Качество контента и Модерация (Quality Score)
- **Индекс качества объявления (Quality Score):**
  - Факторы оценки: количество фото (не менее 5), заполненность ключевых полей (площадь, этаж, цена), уникальность текста описания.
  - Влияние на выдачу: объявления с высоким Quality Score показываются выше при прочих равных.
- **Правила модерации:** автоматическая проверка фото на дубликаты, проверка цены на соответствие рыночному интервалу (защита от фейков).

### 4. Ликвидность и Отклики (Liquidity Loop)
- **Метрика отклика (Reply Rate):** % объявлений, по которым покупатель совершил целевое действие (кликнул «Показать телефон» / написал в чат) в течение 48 часов после публикации.
- **Time-to-Reply (TTR):** медианное время ответа продавца на сообщение покупателя в чате платформы.
- **Сценарии повышения ликвидности:** [автоматические подсказки продавцу снизить цену, если по объявлению нет звонков > 7 дней].

### 5. Безопасность сделки и Защита от фрода (Trust & Safety)
- **Защита номера (Маскирование):** подмена реального номера телефона временным номером платформы для защиты от спамеров и парсеров.
- **Проверка продавца (Verification):** значки «Документы проверены» (по паспорту/Реквизитам компании через Госуслуги / Т-ID / Сбер ID).
- **Безопасная сделка:** удержание средств покупателя на платформе до момента подтверждения получения товара (для товарных вертикалей).

### 6. Ключевые метрики classifieds-платформы
- **View-to-Contact Conversion:** конверсия из просмотра объявления в нажатие кнопки «Показать телефон» или «Написать».
- **ARPU по продавцам:** средний доход с одного подающего объявления пользователя за счет платных услуг и листингов.
- **Liquidity Rate:** доля активных объявлений, по которым была совершена сделка (или получен отклик) до истечения срока публикации.
```

## Правила

- Запрещай классифайды без защиты от спама и дублей. Парсинг чужих баз и массовый перезалив объявлений мгновенно превратят платформу в свалку нерелевантного контента.
- Требуй проработки механизмов верификации авторов (KYC). Значки проверенного профиля через Госуслуги/ID банков — главный фактор доверия покупателей на классифайдах сегодня.
- Монетизация лидогенерации (Pay-per-Lead) должна содержать правила компенсации фейковых лидов. Если риелтор/исполнитель платит за контакт, а пользователь не отвечает или является ботом — платформа должна автоматически вернуть деньги за лид (флоу арбитража).
- Пиши на русском языке.

# Аудит оформления заказа (Checkout Audit)

Создать детальный анализ и план оптимизации воронки оформления заказа (чекаута). Помогает продакту повысить конверсию в оплату, сократить процент брошенных корзин (cart abandonment), снизить трение (friction) при вводе данных и правильно интегрировать современные платежные инструменты.

## Процесс

1. **Разложи воронку чекаута на шаги.** (Корзина → Ввод контактов → Выбор доставки → Выбор оплаты → Экран подтверждения → Успешная оплата).
2. **Определи точки отсева (Drop-off) и барьеры.** Где пользователи уходят (например, неожиданная стоимость доставки, обязательная регистрация, сложная форма ввода адреса)?
3. **Проанализируй платежные методы (Payment Mix).** Достаточно ли способов оплаты? Интегрированы ли бесшовные методы (СБП - Система быстрых платежей, SberPay, Tinkoff Pay, Яндекс Пэй, BNPL/рассрочка «Долями»)?
4. **Оцени трение при вводе данных (Data Entry Friction).** Сколько полей формы заполняет пользователь? Есть ли подсказки, автоподстановка адресов (ФИАС/Дадата)?
5. **Сформулируй UX-рекомендации по оптимизации.** One-page checkout vs. Multi-step checkout, гостевой чекаут без обязательной регистрации, отображение итоговой суммы на каждом шаге.
6. **Сохрани вывод** в текущей рабочей директории как `checkout-audit-[контекст].md`.

## Формат вывода

```
## Аудит воронки оформления заказа: [Название продукта]

### 1. Карта воронки и Точки отсева (Funnel & Drop-offs)
- **Текущая конверсия воронки:** [из корзины в успешную оплату, %]
- **Анализ шагов воронки:**

| Шаг воронки | Конверсия шага | Доля ушедших (Drop-off) | Ключевая причина отсева на шаге |
|-------------|----------------|-------------------------|----------------------------------|
| 1. Корзина | [X%] | [Y%] | [например, нет информации о стоимости доставки] |
| 2. Доставка | [X%] | [Y%] | [например, сложная форма выбора ПВЗ на карте] |
| 3. Оплата | [X%] | [Y%] | [например, ошибка при вводе реквизитов карты] |
| 4. 3D-Secure | [X%] | [Y%] | [например, не пришла СМС с кодом от банка] |

### 2. Анализ платежного трения (Payment Friction & Mix)
- **Набор платежных методов:**
  - *Бесшовные (СБП, SberPay, Tinkoff Pay):* [интегрированы/нет, доля в транзакциях]
  - *Рассрочки / BNPL:* [есть ли оплата частями, на каких чеках работает]
  - *Оплата картой (ручной ввод):* [удобна ли форма ввода, есть ли автоопределение платежной системы]
- **Инфраструктурные ошибки оплаты:** доля ошибок эквайринга (технические сбои, таймауты банка, отказы по 3DS).

### 3. Оценка сложности форм (Form & Data Friction)
- **Количество полей ввода:** [общее количество обязательных и необязательных полей]
- **Автозаполнение:** [используется ли автоподстановка ФИО, автоопределение города по IP, подсказки адресов по ФИАС/КЛАДР]
- **Регистрация:** [обязательна ли регистрация для покупки — главный барьер, или есть покупка в один клик/гостевой доступ]

### 4. Рекомендации по оптимизации UX/UI (Quick Wins)
- **Гостевой чекаут:** разрешить покупку без пароля и подтверждения почты, создавать аккаунт автоматически в фоне после оплаты.
- **Оптимизация доставки:** авторасчет стоимости и сроков доставки до выбора конкретного ПВЗ/адреса.
- **Упрощение формы:** скрыть необязательные поля (например, индекс, второй телефон) под спойлер.
- **Платежные кнопки на первом экране:** вынос кнопок СБП и быстрых оплат прямо на экран корзины для пропуска шагов ввода данных.

### 5. Метрики эффективности чекаута
- **Checkout Conversion Rate:** отношение оплаченных заказов к количеству переходов в чекаут.
- **Cart Abandonment Rate:** % пользователей, которые положили товар в корзину, но не завершили покупку.
- **Payment Success Rate (PSR):** % успешных транзакций от всех попыток оплаты (качество работы эквайринга).
- **Average Order Value (AOV):** влияние ввода рассрочек/BNPL на средний чек (обычно +20-30%).
```

## Правила

- Запрещай требовать обязательную регистрацию перед покупкой. Это убивает до 20-30% конверсии. Требуй внедрения гостевого чекаута (guest checkout) по умолчанию.
- Подсказки адресов обязательны. Ввод индекса, области и района вручную на мобильных устройствах — это критический дефект UX.
- Быстрые платежные методы (СБП, SberPay, Tinkoff Pay) должны быть выведены наверх. Ручной ввод 16 цифр карты на телефоне — это устаревший паттерн, который снижает конверсию.
- Пиши на русском языке.

# Программы лояльности и CRM в ритейле (loyalty-crm)

Спроектировать механику программы лояльности и CRM-кампаний персональных предложений для ритейл-сетей, e-grocery или FMCG-сервисов. Скилл помогает продакту настроить правила начисления бонусов, логику сегментации пользователей по RFM (Recency, Frequency, Monetary), спроектировать триггерные воронки возврата клиентов и рассчитать финансовую эффективность программы (Uplift выручки vs. маржинальные потери на скидки).

## Процесс

1. **Определи тип программы лояльности.**
   - *Прямой кэшбэк/бонусы:* % возврата баллами на карту лояльности.
   - *Многоуровневая (Tiered):* уровни статуса (Серебряный, Золотой) в зависимости от трат за прошлый месяц.
   - *Подписочная (Paid loyalty):* платный бандл подписки для бесплатной доставки и повышенного кэшбэка (например, X5 Пакет).
2. **Спроектируй RFM-сегментацию и логику CRM.** Как разделяем базу:
   - *Спящие (Churn risk):* давно не покупали, высокий средний чек. Механика: реактивация повышенным кэшбэком на 3 дня.
   - *Ядро (Loyal):* покупают часто, стабильный чек. Механика: кросс-продажи (cross-sell) сопутствующих категорий.
3. **Опиши триггерные коммуникации (CRM-карту).** В каких точках шлем push/email/SMS: брошенная корзина, годовщина регистрации, падение частоты визитов ниже исторического среднего.
4. **Рассчитай экономику лояльности.**
   - *Redemption Rate:* % списанных баллов от начисленных (целевой показатель 60-80%).
   - *Margin Erosion (Размытие маржи):* сколько чистой маржи мы теряем из-за предоставления скидок тем, кто купил бы и без них.
   - *Incremental Revenue (Uplift):* дополнительная выручка от изменения поведения пользователей.
5. **Сохрани вывод** в текущей рабочей директории как `loyalty-spec-[название-продукта].md`.

## Формат вывода

```
## Стратегия лояльности и CRM: [Название бренда]

### 1. Архитектура программы лояльности
- **Формат программы:** [Многоуровневая бонусная программа / Платная подписка лояльности]
- **Базовые правила начисления:** [например, 1% баллами за любые покупки, 5% в трех любимых категориях].
- **Правила списания:** [1 балл = 1 рубль. Оплата баллами до 50% чека. Срок жизни баллов — 180 дней].

### 2. RFM-сегментация и CRM механики

| Сегмент клиентов | RFM-критерии | Целевое действие | Персональное предложение (CRM оффер) | Канал доставки |
|------------------|--------------|-------------------|--------------------------------------|----------------|
| **1. Лояльное ядро** | R <= 7 дней, F >= 4 раз/мес | Повышение среднего чека | Скидка 10% на сопутствующую категорию при чеке от 1500 руб. | In-app push |
| **2. Риск оттока** | R > 30 дней, F ранее высокая | Возврат в приложение | «Дарим 300 баллов на карту лояльности. Успейте списать за 5 дней» | SMS / Viber |
| **3. Новички** | R <= 3 дня, F = 1 покупка | Вторая покупка (активация) | Кэшбэк 20% баллами на вторую покупку в течение недели | Push + Email |

### 3. CRM Триггерная карта (Trigger Communications)
- **Триггер: Брошенная корзина в E-grocery:**
  - *Условие:* товары добавлены в корзину, но заказ не оформлен в течение 2 часов.
  - *Действие:* отправка Push с напоминанием: «Ваша корзина ждет вас. Доставим за 30 минут!».
- **Триггер: Падение частоты (Frequency drop):**
  - *Условие:* пользователь не делает заказ в течение периода, превышающего его средний интервал покупок на 50%.
  - *Действие:* отправка персональной скидки на категорию «Любимые товары».

### 4. Финансовые метрики лояльности
- **Incremental Margin (Дополнительная маржа):** целевой рост маржинального дохода от участников лояльности относительно контрольной группы (A/B-тест без скидок).
- **Redemption Rate:** доля списанных баллов.
- **Margin Erosion Rate:** % транзакций со скидкой, совершенных лояльными пользователями без триггера (потери маржи). Должен удерживаться < 15%.
```

## Правила

- Запрещай запускать скидки «на всех» без контрольной группы (Holdout Group). Без выделенной когорты пользователей (обычно 5-10%), которая не получает никаких писем и скидок, невозможно доказать окупаемость CRM-кампаний (вы не будете знать, купили ли люди из-за рассылки или просто потому, что проголодались).
- Обязательно вводи ограничение на срок жизни баллов лояльности. Без этого у вас накопится огромный балансовый долг перед клиентами (невыплаченные обязательства по баллам).
- Пиши на русском языке.

# Операции дарксторов (dark-store-ops)

Оптимизировать продуктовые процессы склада закрытого типа (dark store) и доставки в e-grocery / экспресс-ритейле. Скилл помогает продакту настроить динамическое управление слотами доставки, оптимизировать время сборки (picking time) за счет алгоритмов раскладки товаров и маршрутизации сборщика, контролировать SLA доставки курьерами и рассчитывать юнит-экономику одного заказа/слота.

## Процесс

1. **Спроектируй управление слотами (Slot Management).**
   - *Интервальная доставка:* деление дня на слоты (например, 10:00-12:00). Как закрывать слоты при превышении емкости сборщиков/курьеров?
   - *Экспресс-доставка:* доставка за 15-30 минут. Как динамически менять зону доставки (радиус обслуживания даркстора) в зависимости от погоды, пробок или нехватки курьеров?
2. **Оптимизируй процесс сборки (Picking Optimization).**
   - Алгоритм группировки товаров по зонам хранения (сухая зона, фреш, заморозка, химия).
   - Маршрутизация сборщика по складу (минимизация шагов, обход склада змейкой без возвратов).
   - Метрики: Pick Time per Item (время сборки одной позиции в секундах, норма < 30 сек).
3. **Разработай контроль SLA доставки.** Оценка времени на сборку, передачу курьеру, путь курьера. Логика автоназначения заказов (batching — объединение 2-х заказов одному курьеру).
4. **Опиши юнит-экономику слота/заказа.** Доход с чека + плата за доставку vs. себестоимость товаров + ФОТ сборщика + ФОТ курьера + списания испорченного товара (fresh loss).
5. **Сохрани вывод** в текущей рабочей директории как `dark-store-[название-продукта].md`.

## Формат вывода

```
## Оптимизация операций даркстора: [Название проекта]

### 1. Управление емкостью и Зонами доставки (Dynamic Capacity)
- **Модель доставки:** [Экспресс за 15-30 мин / Интервальные слоты]
- **Алгоритм управления нагрузкой (Surge Strategy):**
  - При нехватке курьеров: автоматическое уменьшение радиуса зоны доставки с 2 км до 1.2 км.
  - При высокой нагрузке на сборку: увеличение прогнозируемого времени доставки в приложении (ETA) на +15 минут.

### 2. Алгоритм и Метрики сборки (Picking Engine)
- **Логика сборки:** Мультисборка (один сборщик собирает до 3 заказов одновременно, если они содержат мало позиций).
- **Схема обхода склада (Routing):** Генерация оптимального пути сборщика в приложении сборщика на основе товарного соседства (начинаем с тяжелых/крупных товаров внизу полок, заканчиваем зоной Fresh и заморозкой).
- **Целевые метрики сборки:**
  - *Pick Time per Item:* < 20 секунд.
  - *SLA сборки заказа:* < 8 минут на заказ из 15 позиций.
  - *Доля ошибок сборки (Pick Accuracy):* > 99.8% (контроль через сканирование штрихкодов сканером ТСД).

### 3. Курьерская доставка и Batching
- **Алгоритм объединения заказов (Batching):** склеивание двух заказов одному курьеру, если их гео-точки находятся в одном направлении (угол < 30 градусов от даркстора) и общий вес < 15 кг.
- **SLA доставки:** < 20 минут с момента передачи заказа курьеру.

### 4. Юнит-экономика одного заказа (Order Economics)
Расчет прибыльности среднего заказа:

| Элемент экономики | Значение | Доля от чека | Описание |
|-------------------|----------|--------------|----------|
| **Средний чек (AOV)** | 1 800 руб. | 100% | Стоимость корзины клиента |
| **Себестоимость (COGS)** | 1 260 руб. | 70% | Закупка товаров (Маржа 30%) |
| **Стоимость сборки (Picking Cost)** | 60 руб. | 3.3% | ФОТ сборщика за 8 минут работы |
| **Стоимость доставки (Delivery Cost)**| 180 руб. | 10% | Выплата курьеру за доставку |
| **Списания / Усушка (Fresh Loss)** | 36 руб. | 2% | Испорченные продукты, овощи/фрукты |
| **Чистая маржа слота (Order Contribution)** | **+264 руб.** | **+14.7%** | Прибыль до учета аренды даркстора |
```

## Правила

- Запрещай сборщикам собирать заказы без сканирования штрихкодов (ТСД). Сборка «на глаз» ведет к пересорту (клиент получает бананы вместо яблок), росту возвратов и падению лояльности.
- Всегда закладывай списания фреш-категории (Fresh Loss / Waste) в юнит-экономику даркстора. E-grocery без списаний не существует; удержание списаний в пределах 2-4% от выручки — отличный продуктовый результат.
- Пиши на русском языке.

# Управление возвратами (returns-management)

Оптимизировать процессы возврата товаров (Reverse Logistics) в e-commerce, маркетплейсах или ритейле. Скилл помогает продакту спроектировать удобный для покупателя интерфейс оформления возврата (Returns UX), внедрить механики защиты от мошенничества (подмена товаров, возврат ношеной одежды), организовать логистику товара обратно на склад/ПВЗ и минимизировать финансовые потери бизнеса на логистику.

## Процесс

1. **Спроектируй пользовательский путь возврата (Returns UX).**
   - Как клиент заявляет о возврате? (Заявка в приложении -> выбор причины -> загрузка фото дефекта -> выбор способа возврата: курьер, ПВЗ, почта).
   - Метрика: Time-to-Refund (время от отправки товара до возврата денег клиенту).
2. **Разработайте правила защиты от фрода (Anti-Fraud).**
   - *Подмена товара (Item swapping):* проверка соответствия штрихкода/бирки на ПВЗ при приеме. Фотофиксация товара сотрудником ПВЗ.
   - *Гардеробные мошенники (Wardrobing):* возврат ношеных вещей. Требования к сохранению товарного вида, ярлыков и пломб.
   - *Черный список:* ограничение возможности возврата без проверки для клиентов с аномально высокой долей возвратов (Return Rate > 80%).
3. **Опиши логистику обратного потока (Reverse Logistics).** Путь товара: ПВЗ -> Сортировочный центр -> Склад брака / Склад перепродажи (восстановление упаковки, химчистка, уценка).
4. **Рассчитай юнит-экономику возврата.** Стоимость обратной логистики + ФОТ сотрудника ПВЗ + ФОТ приемщика на складе + уценка товара.
5. **Сохрани вывод** в текущей рабочей директории как `returns-spec-[название-продукта].md`.

## Формат вывода

```
## Спецификация процесса возвратов: [Название бренда]

### 1. Пользовательский путь оформления возврата (Returns UX)
- **Механика оформления:** Заявка в личном кабинете покупателя.
  - Покупатель выбирает товар из истории заказов, указывает количество и выбирает причину возврата из справочника.
  - Обязательное условие: для причин «Брак» или «Неполный комплект» необходимо загрузить 2-3 фотографии и короткое видео.
- **Способы передачи:** Сдача в любой фирменный ПВЗ бесплатно или вызов курьера за 199 руб. (вычитается из суммы возврата).

### 2. Защита от фрода и мошенничества (Anti-Fraud Gateways)

| Категория риска | Правило валидации | Действие на ПВЗ / Складе |
|-----------------|-------------------|--------------------------|
| **Подмена бренда** | Сопоставление веса коробки и штрихкода на оригинальной бирке (RFID-метки) | Сотрудник ПВЗ сканирует метку; при несовпадении система блокирует кнопку «Принять» |
| **Ношеные вещи** (Wardrobing) | Визуальный осмотр на отсутствие следов носки, запаха парфюма, наличие пломбы фабрики | Отказ в приеме возврата на месте с выдачей бланка мотивированного отказа |
| **Систематический фрод** | Расчет индивидуального Return Rate клиента. Если Return Rate > 75% при покупках от 10 товаров | Отключение опции «Бесплатный возврат». Доставка возврата только за счет клиента |

### 3. Обратная логистика и Сортировка (Reverse Logistics)
- **Точка входа:** ПВЗ принимает товар и упаковывает в сейф-пакет.
- **Сортировочный центр:** Распределение товаров на 3 потока:
  - *Поток A (Годен к продаже):* товар в идеальном состоянии, возвращается на активные полки склада.
  - *Поток B (Уценка/Переупаковка):* требуется замена коробки или мелкий ремонт. Продается со скидкой в разделе «Уценка».
  - *Поток C (Брак/Утилизация):* списание и возврат поставщику.

### 4. Юнит-экономика возврата (Return Cost Analysis)
- **Return Rate (целевой):** [например, 22% в категории Одежда].
- **Структура затрат на один возврат:**
  - *Транспортировка ПВЗ -> Склад:* 120 руб.
  - *ФОТ приемки и сортировки:* 40 руб.
  - *Амортизация товара (потеря стоимости из-за повреждения упаковки):* в среднем 10% от COGS (около 150 руб.).
  - *Итого себестоимость одного возврата:* **310 руб.** (эти затраты должны учитываться в цене товара как резерв на возвраты).
```

## Правила

- Не допускай автоматический безусловный возврат денег (instant refund) до момента физического получения и проверки товара на ПВЗ или складе для дорогих категорий (> 5 000 руб.). Это создает идеальные условия для мошенников, которые будут отправлять пустые коробки.
- ПВЗ-интерфейс приема возвратов должен содержать пошаговый чеклист проверки товара с обязательным фотофиксацией состояния.
- Пиши на русском языке.

# Доставка последней мили и ПВЗ (Last-Mile)

Создать детальную спецификацию требований для логистического продукта последней мили (доставка курьерами, управление ПВЗ, логистические агрегаторы типа СДЭК, Яндекс Доставка). ## Процесс

1. **Определи модель доставки.** (Интервальная доставка в слоты, экспресс-доставка за 15-30 минут, или самовывоз из ПВЗ/постаматов).
2. **Спроектируй процесс назначения заказов на курьеров (Dispatching).** Как распределяются заказы (вручную диспетчером, автоматически алгоритмом группировки по гео-зонам, или курьеры сами выбирают заказы в пуле)?
3. **Опиши требования к приложению курьера (Courier App).** Построение маршрута (навигатор), статусы заказа (принял, приехал, вручил), логика подтверждения вручения (код из СМС, фото двери/паспорта, оплата при получении).
4. **Спроектируй интерфейс пункта выдачи заказов (ПВЗ Portal).** Приемка машины с центрального склада (сканирование штрихкодов), раскладка по ячейкам хранения, выдача клиенту по штрихкоду, оформление возвратов.
5. **Опиши UX отслеживания для покупателя (Customer Tracking UX).** Отображение курьера на карте в реальном времени, динамический расчет времени прибытия (ETA), связь с курьером без раскрытия номеров (маскирование).
6. **Сохрани вывод** в текущей рабочей директории как `last-mile-spec-[название-продукта].md`.

## Формат вывода

```

### 1. Операционная модель и Бизнес-цели
- **Формат доставки:** [например, доставка из даркстора до двери курьером на электровелосипеде, SLA < 20 минут]
- **Схема работы ПВЗ:** [собственная сеть / франшиза / партнерские пункты]
- **Основная цель:** [например, снижение Cost per Delivery на 15% за счет автогруппировки заказов в мультизаказы]

### 2. Алгоритм диспетчеризации и Маршрутизация (Courier Routing)
- **Автоматическое назначение (Dispatching):** алгоритм подбора оптимального курьера на заказ (учитывает статус курьера, его тип транспорта, вместимость сумки и расстояние до склада/клиента).
- **Мультизаказ (Batching):** правила объединения нескольких заказов в один маршрутный лист курьера (например, курьер берет максимум 3 заказа, если их адреса находятся в радиусе 1.5 км).

### 3. Приложение курьера (Courier App Capabilities)
- **Экран маршрута:** интеграция с картами/навигатором, отображение точек А (забор) и Б (вручение) с комментариями («вход со двора», «код домофона»).
- **Статусная модель:** переход статусов [Назначен -> На складе -> В пути -> На месте -> Доставлен / Отменен].
- **Флоу закрытия заказа:** подтверждение доставки (сканирование штрихкода посылки, ввод OTP-кода от клиента, или фотофиксация бесконтактной доставки).

- **Приемка отправлений:** массовое сканирование посылок при разгрузке курьерской машины, фиксация повреждений/недостач.
- **Адресное хранение:** сканирование посылки -> сканирование полки/ячейки (чтобы быстро находить посылку при выдаче).
- **Выдача по QR/Штрихкоду:** клиент показывает штрихкод из приложения -> сотрудник сканирует его -> система подсвечивает номера ячеек с посылками клиента.
- **Частичный выкуп и Возвраты:** флоу примерки одежды, отказ от части позиций, перерасчет чека и отправка возврата на склад.

### 5. Трекинг для клиента (Customer Tracking UI)
- **Live Tracking:** отображение местоположения курьера на карте, когда заказ переходит в статус «В пути».
- **Динамический ETA:** обновление времени доставки в зависимости от пробок и движения курьера по маршруту.
- **Безопасная связь:** чат и звонки курьеру через VOIP/телефонию банка/платформы без раскрытия реального номера телефона клиента.

### 6. Метрики эффективности Last-Mile
- **On-Time Delivery (OTD):** % заказов, доставленных вовремя без нарушения SLA / временного слота.
- **Cost per Delivery (CPD):** себестоимость одной доставки (ФОТ курьера, амортизация, накладные расходы).
- **Утилизация курьера (Trips per Hour):** количество доставленных заказов в час на одного активного курьера.
- **Order-to-Delivery Time:** общее время от сборки заказа до передачи клиенту.
```

## Правила

- Оспаривай системы доставки без автоматического распределения заказов (batching/dispatching). Ручное распределение диспетчером не масштабируется на объемах выше 50 заказов в день и ведет к простоям курьеров.
- Требуй наличия адресного хранения в ПВЗ. Если в требованиях к ПВЗ нет привязки «посылка — ячейка», время выдачи заказа вырастет в разы, создавая очереди.
- Защита персональных данных курьера и клиента обязательна. Прямые звонки на личные номера телефонов должны быть запрещены — используй маскирование номеров через АТС.
- Пиши на русском языке.

# Управление промо-механиками (Promo Engine)

Создать продуктовую спецификацию требований для системы управления промо-акциями, скидками и промокодами (Promo Engine). Помогает продакту спроектировать гибкие механики лояльности, рассчитать окупаемость акций, избежать критической деградации маржи продукта и защитить платформу от злоупотреблений скидками (promo abusing).

## Процесс

1. **Определи тип промо-акции.** (Прямая скидка на товар, скидка по промокоду, механики 1+1=3, скидка на сумму корзины, кэшбэк баллами лояльности).
2. **Спроектируй правила применения (Promo Rules).** Каковы ограничения? (Минимальная сумма заказа, применимость только к определенным категориям, несовместимость с другими акциями, лимит на количество использований одним пользователем).
3. **Рассчитай влияние на экономику (Margin Impact).** Как акция влияет на маржинальность? (Расчет деградации маржи, оценка инкрементального объема продаж, необходимого для выхода акции в плюс).
4. **Опиши механику персонализации.** Как подбираются скидки для разных сегментов? (Скидки для новичков на первую покупку, реактивационные скидки для оттекающих клиентов, персональный кэшбэк на любимые категории).
5. **Спроектируй защиту от фрода (Anti-Fraud).** Как защитить систему от создания мультиаккаунтов ради приветственной скидки, перепродажи товаров оптовиками под видом розничных акций.
6. **Сохрани вывод** в текущей рабочей директории как `promo-spec-[название-акции].md`.

## Формат вывода

```
## Спецификация промо-механики: [Название акции]

### 1. Концепт акции и Бизнес-цели
- **Тип промо-механики:** [например, скидка 500 руб. на первый заказ от 2000 руб. по промокоду]
- **Основная цель:** [привлечение новых клиентов (Acquisition) / реактивация старых / увеличение среднего чека (AOV)]
- **Целевой сегмент:** [например, пользователи без совершенных заказов в системе]

### 2. Правила валидации промокода (Promo Logic)
- **Условия активации:**
  - Минимальный чек заказа: [X] руб.
  - Ограничения по каталогу: [например, не применяется на электронику, алкоголь и товары со значком «финальная цена»].
  - Пересечение скидок: [промокод не суммируется с другими промокодами и баллами лояльности].
  - Время жизни (TTL): срок действия акции.
- **Ограничения по лимитам:** общий бюджет акции (например, не более 1 000 000 руб. суммарной скидки на всю кампанию) и лимит на одного пользователя.

### 3. Экономика акции (Margin degradation & Incrementality)
- **Базовая маржа:** [X%]
- **Маржа с учетом скидки:** [Y%]
- **Расчет инкрементальности:** [на сколько процентов должны вырасти продажи (conversion lift), чтобы покрыть затраты на скидку и сохранить общую валовую прибыль].
- **CAC / ROI акции:** прогноз стоимости привлечения нового покупателя с учетом стоимости промокода.

### 4. Интеграция с лояльностью и Персонализация
- **Начисление кэшбэка:** сколько баллов лояльности получает пользователь за заказ с промокодом.
- **Triggered Promos (Триггерные промо):** автоматическая отправка персонального промокода в пуш/email при наступлении события (например, брошенная корзина 24 часа назад, день рождения клиента).

### 5. Защита от злоупотреблений (Anti-Fraud Rules)
- **Борьба с реферрал-фродом:** [проверка связей аккаунтов по номеру телефона, устройству (device ID), отпечатку браузера (fingerprint), банковской карте].
- **Блокировка мультиаккаунтинга:** запрет на применение промокода «на первую покупку», если с этого устройства или на этот адрес доставки уже были заказы.
- **Лимиты на частоту запросов (Rate Limiting):** защита API чекаута от перебора промокодов (брутфорс).

### 6. Метрики эффективности промо-кампании
- **Redemption Rate:** % пользователей, применивших промокод, от числа тех, кто его получил.
- **Incremental Sales/GP:** объем дополнительных продаж и валовой прибыли, сгенерированных акцией.
- **AOV Lift:** прирост среднего чека по заказам с промокодом в сравнении с контрольной группой.
- **Share of Promo Sales:** доля заказов со скидками в общем объеме продаж платформы (опасен рост > 30-40%, когда пользователи привыкают покупать только по акциям).
```

## Правила

- Запрещай запускать промо-кампании без расчета деградации маржи. «Сделаем скидку 20%, чтобы вырастить продажи» — путь к кассовому разрыву. Продакт обязан рассчитать необходимый рост конверсии/чека для окупаемости.
- Защита от фрода — обязательная часть спецификации. Приветственные промокоды без проверки по Device ID и номеру телефона будут моментально выгреблены профессиональными абузерами и ботами.
- Суммирование скидок должно быть ограничено. Спецификация должна четко описывать приоритет применения скидок (сначала скидка на товар, потом промокод, потом списание баллов).
- Пиши на русском языке.

# Управление цепочками поставок в ритейле (Supply Chain)

Создать детальную спецификацию требований для системы управления запасами, автозаказа или складской логистики в ритейле и e-groceries (аналоги Самоката, Яндекс Лавки, Х5, Магнита). Помогает продакту автоматизировать процесс пополнения запасов, минимизировать списания скоропортящихся товаров (fresh) и максимизировать доступность товаров для покупателей.

## Процесс

1. **Определи тип склада/точки.** (Даркстор экспресс-доставки, центральный распределительный центр (РЦ), или полка физического супермаркета).
2. **Спроектируй модель автозаказа (Auto-Replenishment).** Каков алгоритм расчета объема заказа? (Прогноз спроса на основе исторических продаж + страховой запас (safety stock) + учет сезонности, акций и погоды).
3. **Опиши логику управления свежестью (Fresh & Expiry Date).** Как система управляет товарами с коротким сроком годности? (Механика FEFO — First Expired, First Out; автоматическое снижение цены перед истечением срока; списание просрочки).
4. **Спроектируй метрику доступности товара (OSA - On-Shelf Availability).** Как отслеживать, что товар реально доступен для покупки? Как выявлять «виртуальные остатки» (когда в системе товар числится, но по факту украден или потерян на складе).
5. **Определи ключевые метрики логистики.** Out-of-Stock rate (доля упущенных продаж), Write-off rate (доля списаний в выручке), оборачиваемость запасов (Inventory Turnover), точность прогноза спроса (Forecast Accuracy).
6. **Сохрани вывод** в текущей рабочей директории как `supply-chain-spec-[название-продукта].md`.

## Формат вывода

```
## Спецификация Supply Chain: [Название продукта]

### 1. Архитектура цепочки поставок и Контекст
- **Тип логистического узла:** [например, даркстор экспресс-доставки 15 минут, 2000 SKU]
- **Схема поставщиков:** [прямые поставки от дистрибьюторов на даркстор / кросс-докинг через РЦ]
- **Ключевой вызов:** [баланс между доступностью товара (OSA) и уровнем списания просрочки (write-offs)]

### 2. Алгоритм автозаказа (Auto-Replenishment Logic)
- **Формула заказа (Reorder Point):** `Заказ = Прогноз спроса на период плеча поставки + Страховой запас - Текущий остаток - Товар в пути`.
- **Факторы прогнозирования:** [какие переменные учитывает модель: тренд продаж, день недели, праздники, погода (влияет на спрос воды/мороженого), локальные промо-акции].
- **Интервал заказа (Lead Time):** плечо поставки (время от заказа у поставщика до приемки на складе).

### 3. Специфика категории Fresh и Контроль сроков годности
- **Управление сборкой (FEFO):** требования к интерфейсу сборщика (приложение darkstore picker) для выбора товаров с наименьшим сроком годности в первую очередь.
- **Динамический уценочный двигатель (Markdown Engine):** автоматическое создание скидки (например, -30%) на товары, у которых осталось менее 20% срока годности.
- **Контроль списаний:** автоматическая блокировка продажи товара на кассе/в приложении за 1 день до истечения срока годности.

### 4. Контроль доступности на полке (OSA & Virtual Stock)
- **Детекция виртуальных остатков:** алгоритм выявления расхождений (например, если остаток товара в системе > 3 шт., но за последние 2 дня не было ни одной продажи при высоком спросе — система ставит задачу сотруднику склада провести инвентаризацию).
- **Out-of-Stock (OOS) логика:** автоматическое скрытие товара из витрины приложения при нулевом фактическом остатке, предложение аналогов пользователю.

### 5. Метрики эффективности цепи поставок (Logistics KPIs)
- **OSA (On-Shelf Availability):** % времени, когда товар из топ-матрицы был доступен для покупки.
- **Write-off Rate:** стоимость списанных товаров (истекший срок, брак) к общему товарообороту.
- **Inventory Turnover (Оборачиваемость):** средний срок нахождения товара на складе в днях (целевое значение для свежих категорий < 2-3 дней).
- **Forecast Accuracy (WAPE):** точность прогноза спроса системы автозаказа.
```

## Правила

- Запрещай ручной заказ товаров директорами складов/дарксторов по умолчанию. Человеческий фактор ведет к перезатаренности складов или отсутствию товаров. Процесс должен быть автоматическим (автозаказ), с возможностью ручной корректировки только при форс-мажорах.
- Требуй описания логики FEFO (First Expired, First Out) для сборки. Сборка по принципу LIFO («взял то, что лежит сверху/ближе») ведет к порче товаров в глубине полки.
- Обязательно учитывай логику виртуального стока. В e-grocery разница между базой данных и фактом на полке ведет к тому, что сборщик отменяет позиции в заказе клиента, снижая NPS.
- Пиши на русском языке.

# Аудит оптимизации конверсии (CRO Audit)

Провести профессиональный аудит конверсии (Conversion Rate Optimization) посадочной страницы, экрана регистрации или воронки онбординга. Помогает продакту выявить барьеры, мешающие пользователям совершить целевое действие, проанализировать контент по ключевым психологическим факторам и составить приоритетный список изменений для A/B-тестирования.

## Процесс

1. **Определи целевую страницу и ключевое действие.** (Лендинг продукта, страница тарифов, экран привязки карты. Целевое действие — оставить заявку, купить подписку, зарегистрироваться).
2. **Проведи аудит по фреймворку CRO:**
   - *Clarity (Понятность):* Понятно ли с первых 3 секунд, что предлагает продукт и каков следующий шаг (Call to Action - CTA)?
   - *Relevance (Релевантность):* Соответствует ли контент ожиданиям пользователя (после клика на рекламу)?
   - *Friction (Трение):* Насколько сложно выполнить целевое действие (лишние поля форм, долгая загрузка, сложный выбор)?
   - *Trust (Доверие):* Есть ли на странице доказательства безопасности и надежности (сертификаты, логотипы банков, отзывы клиентов, гарантии)?
   - *Urgency (Срочность):* Стимулирует ли страница принять решение быстрее (ограниченные акции, лимиты мест, таймеры)?
3. **Составь план гипотез для A/B-тестов (Top-5 Quick Wins).** Что конкретно изменить в интерфейсе, текстах или логике для роста конверсии.
4. **Сохрани вывод** в текущей рабочей директории как `cro-audit-[название-страницы].md`.

## Формат вывода

```
## CRO-аудит конверсии страницы: [Название страницы]

### 1. Целевое действие и Текущая конверсия
- **Описание страницы:** [например, посадочная страница B2B-кабинета застройщика]
- **Ключевой Call to Action (CTA):** [кнопка «Зарегистрировать компанию»]
- **Текущая конверсия (CR):** [например, 2.4% из посетителя в регистрацию]

### 2. Детальный аудит по фреймворку CRO
Анализ барьеров и триггеров:
- **Clarity (Понятность):**
  - *Плюсы:* Название компании и логотип хорошо видны.
  - *Минусы:* Главный заголовок размыт («Инновации для рынка недвижимости» вместо конкретного «Автоматизация раскрытия эскроу-счетов»). Кнопка CTA находится ниже первого экрана прокрутки.
- **Relevance (Релевантность):**
  - *Анализ:* Трафик идет с рекламы по запросу «ипотека ИЖС», но страница говорит только про МКД-строительство. Пользователь не находит подтверждения своего запроса.
- **Friction (Трение):**
  - *Анализ:* Форма регистрации содержит 12 обязательных полей (включая ОГРН, БИК банка и личный телефон генерального директора). Нет подсказок при вводе.
- **Trust (Доверие):**
  - *Анализ:* Отсутствуют логотипы банков-партнеров, нет информации о лицензиях и безопасности данных. Раздел «Отзывы» выглядит фальшивым (текст без реальных фото и названий компаний).
- **Urgency (Срочность):**
  - *Анализ:* Нет стимулов принять решение сейчас (акция «Первые 3 сделки бесплатно при подключении до конца недели» не выведена на первый экран).

### 3. План гипотез для A/B-тестирования (Top-5 Quick Wins)

| № | Описание изменения (Что меняем) | Ожидаемый эффект (За счет чего вырастет конверсия) | Сложность реализации (Лоу/Мид/Хай) | Приоритет (ICE Score) |
|---|----------------------------------|----------------------------------------------------|-----------------------------------|-----------------------|
| 1 | Сократить форму регистрации с 12 до 3 обязательных полей (ИНН, Email, Телефон). Остальное заполнять в фоне по ИНН. | Снижение трения (Friction) при вводе данных. | Лоу (интеграция Dadata) | Высокий |
| 2 | Переписать главный заголовок первого экрана под боли ИЖС: «Раскрывайте эскроу-счета ИЖС автоматически за 24 часа». | Повышение релевантности (Relevance) и ясности. | Лоу | Высокий |
| 3 | Поднять кнопку CTA на первый экран и сделать её контрастной. | Повышение видимости целевого действия (Clarity). | Лоу | Высокий |
| 4 | Добавить блок доверия: «Банк-эскроу агент первого уровня» + лицензия ЦБ РФ. | Рост доверия (Trust) в финтех-сегменте. | Лоу | Средний |
| 5 | Добавить таймер/баннер: «Бесплатный тестовый период 30 дней при регистрации до [Дата]». | Создание мягкой срочности (Urgency). | Лоу | Средний |

### 4. Метрики для отслеживания в тестах
- **Conversion Rate (CR):** % посетителей, успешно прошедших регистрацию.
- **Bounce Rate (Показатель отказов):** % пользователей, ушедших со страницы в первые 15 секунд без взаимодействий.
- **Form Drop-off Rate:** % пользователей, которые начали заполнять форму регистрации, но бросили её на полпути.
```

## Правила

- Оспаривай длинные формы ввода данных. Каждое лишнее поле в форме снижает конверсию на 5-10%. Если продакт просит собрать максимум данных на первом шаге — требуй сокращения полей до абсолютного минимума.
- Главный заголовок первого экрана (Hero Section) должен содержать ценностное предложение (Value Proposition), понятное без прокрутки страницы. Никакой размытой корпоративной философии.
- Все гипотезы для A/B-тестов должны быть приоритизированы по фреймворку (ICE/RICE).
- Пиши на русском языке.
